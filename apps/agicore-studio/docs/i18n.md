# Internationalization

The Studio ships English-only at 1.0 but the scaffolding is wired
so contributor translations can land as additional JSON files
without touching component code.

## Adding a string

```tsx
import { t } from '../i18n';

// before
<button title="Save">Save</button>

// after
<button title={t('settings.save')}>{t('settings.save')}</button>
```

1. Add the key to `src/i18n/en.json` with a sensible namespace
   prefix — `settings.foo.bar`, `palette.heading`, etc. Group by
   surface so the JSON stays readable.
2. Replace the hard-coded string in the component.
3. Done. No build step, no extraction tool, no precompiled bundle —
   the JSON is imported at module load.

## Interpolation

Use `{name}` placeholders in the source string and pass values as
the second argument:

```json
"palette.add_button_label": "Add a {kind} node"
```

```tsx
t('palette.add_button_label', { kind: item.paletteLabel })
```

Missing placeholders fall through as the raw `{name}` literal so a
contributor-written translation that forgets a variable degrades
visibly rather than silently.

## Why not a library?

We considered react-intl, i18next, FormatJS. All of them solve
problems we don't have:

- **Pluralization rules** for languages with multiple plural forms.
  English doesn't need them; languages that do can land that work
  alongside their first translation.
- **Async resource loading** for code-split locales. We have one
  locale and ten kilobytes of strings; eagerly loading them at
  module init is fine.
- **Locale negotiation** with the OS / browser. We default to en;
  a locale picker lands when a second locale lands.

A 30-line `t()` helper covers everything we need. The library
upgrade path stays open — if we ever add Japanese with its plural
rules and four numeric formats, we revisit. Until then, the simple
thing.

## Adding a language

For a contributor wanting to ship a translation:

1. Copy `src/i18n/en.json` to `src/i18n/<lang>.json` (e.g. `ja.json`).
2. Translate every value. Keys are stable; values are yours.
3. Add the import + map entry in `src/i18n/index.ts`:

   ```ts
   import en from './en.json';
   import ja from './ja.json';

   const LOCALES: Record<Locale, Record<string, string>> = { en, ja };
   ```

4. Extend the `Locale` type union.
5. Open a PR. We'll wire a locale picker when the second locale
   lands.

## Coverage status

Today we've migrated the highest-traffic surfaces — TitleBar window
controls, NodePalette heading and per-kind add label. The rest of
the components still have inline strings; this is intentional
incrementalism, not negligence. The migration pattern is documented
above; PRs that move components over are welcome.

The closer we get to 1.0, the more migration matters — but a
half-migrated codebase that's clear about what's left beats a
"we'll do i18n later" codebase that ships hard-coded strings and
hopes.
