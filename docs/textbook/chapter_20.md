There is a gap between querying an AI and engineering an AI inference. The gap is not visible in the individual output. A well-crafted ad hoc prompt can produce an output indistinguishable from — and occasionally superior to — the output of a structured production system. The gap is visible only over time, at volume, under quality standards that must be maintained consistently across thousands of outputs produced by practitioners of varying skill, on days of varying attention, against datasets of varying size and quality. The ad hoc query cannot be reproduced exactly. Its quality cannot be measured systematically. Its failures cannot be diagnosed structurally. Its prompt cannot be improved incrementally without risking regression on previously passing cases. It has no audit trail. It produces no run history from which a control chart can be constructed. It is an act of individual craft, which is to say it is irreducibly dependent on the practitioner who performs it. **Craft cannot scale.** The research team that uses ad hoc queries to produce executive briefings is as dependent on its best analyst as a pre-industrial workshop was dependent on its master artisan. The output quality follows the practitioner's quality. When the practitioner leaves, the quality leaves. When the organization needs to produce ten briefings instead of one, it needs ten analysts of equivalent skill, not one analyst with a faster computer. The REASONER declaration closes this gap. It is the mechanism by which structured AI inference becomes an engineered production process rather than a craft practice.

The distinction between exploration and production is the first conceptual boundary that the REASONER makes precise. **Exploratory inference** is what a practitioner does when they are discovering the shape of a problem: testing prompt formulations, exploring model behavior, generating candidate approaches, finding the vocabulary of a domain. Exploratory inference is creative and undisciplined in the productive sense — it benefits from looseness, from trying unexpected angles, from treating model outputs as provocations rather than deliverables. It does not require reproducibility, audit trails, or structured quality assessment. A Jupyter notebook is a perfectly adequate environment for exploratory inference. **Production inference** is what a cognitive pipeline does when it must produce a deliverable that meets a defined quality standard, at volume, with full traceability, day after day. Production inference requires a defined corpus, a structured context assembly process, a formal output contract, a designated model, and a run audit record. The gap between these two kinds of inference is not a matter of prompt quality. It is a matter of architecture. A very good ad hoc prompt is still a craft artifact. A REASONER declaration is an engineering artifact. The REASONER declaration does not improve the quality of any single AI call. It makes the quality of AI calls consistent, measurable, and improvable across the full production run.

The **REASONER** declaration in Agicore is a formal specification of a structured AI inference run. It has five required components. The first is the **USING clause**, which specifies the ENTITY type or types from which the reasoner accumulates its input corpus. The USING clause names a declared ENTITY, and at runtime, the Agicore engine queries the database for all records of that entity type that meet the reasoner's accumulation criteria, assembling them into an ordered context that becomes the AI call's primary input. The second is the **CONSIDERING clause**, which specifies which fields of the accumulated records to include in the assembled context. Not all fields in an entity record are relevant to every inference task. A research synthesis reasoner may need the `findings`, `source`, `confidence`, and `query` fields but not the `created_at`, `operator_id`, or `archive_status` fields. The CONSIDERING clause is a projection: it specifies the semantic surface of each record that the inference will operate on, reducing noise and token consumption simultaneously. The third is the **PRODUCING clause**, which specifies the output contract: the type that the reasoner's output must conform to. In current Agicore implementations, the primary output type is `String`, which carries the inference result as a structured text document whose format is specified by the system prompt. Future implementations may support typed structured output contracts. The fourth is the **MODEL clause**, which designates the specific AI model to use for this inference run. Model designation at the reasoner level enables the multi-model architecture discussed later in this chapter — different reasoners using different models calibrated to their specific cognitive demands. The fifth is the implicit audit obligation: every REASONER run produces a **RUN record** — a database entry that captures the run ID, reasoner name, model used, status, accumulation count, start time, completion time, and output text. The RUN record is not optional and is not configurable. It is a structural consequence of declaring a REASONER.

The **multi-record accumulation pattern** is the feature that most sharply distinguishes REASONER from ad hoc AI queries and deserves extended treatment. An ad hoc query presents the model with whatever context the practitioner chooses to include in the prompt window — a paragraph, a document, a set of documents manually assembled. The assembly is a one-time act performed by the practitioner for that query. There is no systematic process by which the full domain state is surveyed, relevant records selected, and context assembled according to consistent rules. The REASONER's USING and CONSIDERING clauses formalize this assembly process. When the Agicore runtime executes a REASONER, it performs a structured database query against the declared ENTITY type, retrieves the qualifying records, projects each record to the fields specified in the CONSIDERING clause, and assembles the projected records into an ordered context. This context — potentially representing dozens or hundreds of records — becomes the body of the AI call. The AI model reasons over the full accumulated corpus, not over a practitioner-selected subset. The analogy to data analytics is exact: a business intelligence report does not present the analyst's favorite three rows from the sales database — it aggregates the full dataset according to defined criteria and presents the aggregate. The REASONER does the same for AI inference: it aggregates the domain's current state according to declared criteria and reasons over the aggregate. This is why REASONER outputs are called **structured inferences** rather than responses: they are the result of a formally specified reasoning process over a formally assembled corpus, not the result of a conversation.

Consider a concrete REASONER declaration for a research synthesis pipeline:

```
reasoner SynthesizeAnalysis {
  description "Synthesize research findings into executive summary"
  using ResearchRecord
  considering ["query", "findings", "source", "confidence"]
  producing String
  model "claude-opus-4-7"
}
```

At runtime, when this reasoner executes, the Agicore engine queries the database for all `ResearchRecord` entities that have been marked for inclusion in the current synthesis run, projects each record to its `query`, `findings`, `source`, and `confidence` fields, and assembles the projected records into an ordered context. This context is passed to `claude-opus-4-7` along with a system prompt — encoded in the reasoner's operational specification — that instructs the model how to synthesize the accumulated findings into an executive summary meeting the pipeline's quality standard. The model produces the synthesis. The Agicore runtime writes the output to the `SynthesizeAnalysis_runs` table as a RUN record with status `completed`, accumulation count equal to the number of `ResearchRecord` entities assembled, and output text equal to the synthesis. The full run is now a database record. It is auditable, queryable, and comparable to prior runs.

The **RUN record as control chart** is not a metaphor — it is a direct application of SPC principles to structured AI inference. Walter Shewhart's insight was that manufacturing quality could not be improved by inspecting finished products and discarding defects; it could only be improved by understanding the process that produced them, distinguishing between chance variation (inherent in the process) and assignable variation (caused by specific, identifiable factors), and eliminating assignable causes. The same principle applies to REASONER runs. A reasoner whose pass rate varies from 95% to 60% depending on the day has a quality problem that is not visible in any individual run — it is visible only in the run history. The RUN record makes the run history available for SPC analysis: the practitioner can plot pass rate over time, model used, accumulation count, output length, and any quality scores assigned by QC reasoners, and apply Shewhart's chart to detect when the process has shifted out of statistical control. The RUN record is the data that makes REASONER behavior improvable in the same way Shewhart made machining processes improvable — not by reacting to individual failures, but by understanding the distribution of outcomes over many runs and intervening when the distribution shifts.

The **REASONER as QC node** is the architectural symmetry that makes quality assurance in Agicore simultaneously powerful and simple. There is no distinct QC declaration type in Agicore. A QC node is a REASONER whose task — encoded in its system prompt — is to evaluate the output of a preceding REASONER rather than to transform source data. The syntax is identical; the semantics are categorically different. A QC REASONER accumulates RUN records from a preceding processing REASONER using the USING clause, projects the relevant fields — typically `output`, `accumulation_count`, `model`, and `status` — using the CONSIDERING clause, and produces a pass/fail evaluation with diagnostic detail using the PRODUCING clause. The QC reasoner's system prompt specifies the quality criteria against which it evaluates the processing reasoner's output. This is the formal implementation of the QC node architecture established in the cognitive pipeline engineering section: the QC node receives the processing node's output, evaluates it against defined criteria, and produces a binary judgment with a diagnostic message. The only difference between the chapter 13 formulation and the Agicore implementation is that the Agicore formulation requires no special infrastructure — the QC node is a REASONER declaration using the same ENTITY abstraction, the same accumulation pattern, and the same RUN record mechanism as any other reasoner.

```
reasoner ValidateAnalysis {
  description "QC node: verify analysis meets quality criteria"
  using AnalysisRun
  considering ["output", "source_count", "model"]
  producing String
  model "claude-haiku-4-5-20251001"
}
```

Notice that `ValidateAnalysis` uses `claude-haiku-4-5-20251001` — a faster, cheaper model — while `SynthesizeAnalysis` uses `claude-opus-4-7`. This model selection reflects the cognitive demand difference between synthesis and evaluation. Synthesis requires the model to hold a large accumulated corpus in context and produce a coherent, nuanced document that integrates disparate findings. This is among the most cognitively demanding tasks in the AI inference taxonomy. Evaluation requires the model to apply a defined rubric to a single text and produce a structured judgment. This is cognitively demanding in the sense of requiring careful instruction-following and resistance to the model's natural inclination to be generous in evaluation — but it does not require the same breadth of synthesis capability. The cost difference between the two models may be substantial. Applying the Opus-class model to evaluation tasks that Haiku-class models handle adequately is a form of resource waste that compounds across every QC run in every pipeline. The REASONER declaration makes model selection an explicit per-task engineering decision rather than a system-wide default.

**Multi-model reasoner design** is not merely an optimization — it is a reflection of the cognitive heterogeneity of the pipeline's tasks. Four task classes warrant distinct model selection strategies. The first is **high-stakes synthesis**: tasks that require integrating large, diverse corpora into coherent, nuanced outputs — executive strategy documents, comprehensive research syntheses, multi-source analytical narratives. These tasks warrant the highest-capability, highest-cost models. The marginal quality gain from the best available model on synthesis tasks is typically real and often large: the difference between a passable synthesis and an excellent one is frequently traceable to the model's ability to maintain coherent argument structure across a long context. The second is **volume classification**: tasks that require categorizing, tagging, or routing a large number of inputs according to a well-defined taxonomy. These tasks warrant fast, cheap models. The accuracy ceiling for a well-specified classification task is typically high for mid-capability models; the incremental quality gain from upgrading to a high-capability model is small relative to the cost increase. The third is **QC evaluation**: tasks that require applying a defined rubric to an existing output. The cognitive demands here are different from both synthesis and classification — they require careful, criterion-anchored evaluation rather than generative output. Models with strong instruction-following and calibrated reasoning perform well on evaluation tasks; raw synthesis capability is less important. The fourth is **structured extraction**: tasks that require pulling specified fields or patterns from unstructured text according to a formal schema. Extraction tasks may benefit from models fine-tuned on extraction benchmarks, or from smaller models with very specific extraction prompts. The point is that model selection is not uniform across task types, and the REASONER declaration makes this non-uniformity an explicit, documented, compile-time-validated architectural decision.

The **trigger→reasoner→memory loop** is the formal implementation of Principle 8 — Continuous Improvement — at the system level, and its concrete syntax reveals how the three declaration types compose into a learning architecture. The TRIGGER fires a REASONER when a condition is met. The REASONER executes and produces an output, which the Agicore runtime stores in the RUN record table. The Agicore runtime then automatically calls the `store_insight` operation with the RUN record's key fields — a call that writes a semantic memory entry to the knowledge store with namespace, key, value, confidence, source, and TTL fields as established in the knowledge architecture chapter. The memory entry persists for the TTL duration. A subsequent REASONER run — fired by its own trigger when new work arrives — can include a `recall` operation in its context assembly, pulling relevant prior insights from semantic memory and incorporating them into the assembled context before the AI call. The system has learned: the output of a prior REASONER run has influenced the context of a later one. This is not machine learning in the gradient descent sense — the model weights have not changed. It is organizational learning in the Senge sense: the system's behavior has been modified by its accumulated experience, encoded in the semantic memory store and recalled at inference time.

In concrete terms, consider a pipeline that processes competitive intelligence reports. The synthesis reasoner produces an insight: a competitor has accelerated its supply chain localization strategy. The Agicore runtime stores this insight in semantic memory with key `competitor_strategy_shift`, namespace `competitive_intelligence`, confidence 0.82, source `SynthesizeAnalysis_run_4471`, and TTL of ninety days. Three weeks later, a new batch of competitive intelligence is processed. The synthesis reasoner's context assembly begins with the standard record accumulation — new ResearchRecord entities from the current batch. But before passing the assembled context to the model, the Agicore runtime executes the recall operation, querying semantic memory for insights relevant to the current query. The prior insight about supply chain localization is retrieved and appended to the assembled context. The model synthesizes the new batch in the context of the prior insight. Its output is richer, more continuous, and more useful than it would have been if the system had started from scratch. The system has incorporated its prior learning. This is the cognitive production system's implementation of organizational memory: formal, automated, auditable, and bounded by TTL governance that prevents stale insights from persisting indefinitely into contexts where they are no longer valid.

The RUN record also enables a practitioner-facing feedback loop that operates outside the automated pipeline. Because every REASONER execution produces a database record with a full output and a QC status, the practitioner can query the run history and apply their domain judgment to runs that the automated QC system passed. This is the human-in-the-loop mechanism that the Cattle Dog Principle requires: the practitioner decides, the system executes and produces an auditable record, and the practitioner reviews the record to confirm that the system's judgment aligns with theirs. When it does not — when the practitioner identifies a RUN record that passed automated QC but fails their domain standard — they have a specific, concrete artifact to reason about: a run ID, a model, an accumulation count, an output text, and a QC evaluation. They can identify the gap between the QC reasoner's criteria and their own judgment, update the QC reasoner's system prompt to incorporate the missing criterion, and rerun the QC reasoner against prior runs to measure the effect. This is continuous improvement in Shewhart's sense: not reacting to individual failures, but updating the quality standard based on practitioner judgment and measuring the effect on the historical run distribution.

The REASONER declaration is, in the deepest sense, the answer to the inference problem stated at the opening of this chapter. The problem is that unstructured queries produce unstructured outputs whose quality is entirely dependent on the practitioner who formulated them. The REASONER solves this problem not by making any single AI call more reliable — the model is still probabilistic, and the output of any single run is drawn from a probability distribution — but by making the process that governs AI calls formal, documented, reproducible, and improvable. The corpus is specified in the USING clause and assembled deterministically from the domain model. The context is projected by the CONSIDERING clause, removing irrelevant fields and reducing noise. The output contract is specified in the PRODUCING clause, defining what a valid output looks like. The model is designated in the MODEL clause, making model selection an explicit engineering decision. The RUN record captures the full execution history, enabling SPC analysis and practitioner review. None of these mechanisms change the probability distribution of a single AI call. All of them change the distribution of outcomes across thousands of calls — by ensuring that each call operates on the same corpus, with the same context projection, against the same output contract, with the same model, and with a record of the result. The individual AI call remains probabilistic. The REASONER's production behavior is governed by a deterministic specification. The reliability lives in the specification, exactly as the cognitive pipeline architecture established in Part IV requires. The REASONER is the implementation of that requirement at the declaration level.

The practitioner who is new to REASONER design frequently makes a characteristic error: they design the CONSIDERING clause too broadly, including every field in the ENTITY in the assembled context, on the theory that more information is always better for the model. This theory is incorrect in the production context, for reasons that are both empirical and architectural. Empirically, large language models have known attention limitations when presented with very long contexts: information in the middle of a long context receives less reliable attention than information at the beginning or end, a phenomenon documented in the research literature as the "lost in the middle" problem (Liu et al., 2023). A CONSIDERING clause that projects all twenty-three fields of a complex ENTITY into the assembled context for every record in the accumulation set will, for large accumulation counts, produce contexts in which the most important information is buried in the middle of tens of thousands of tokens. The model's output quality degrades not because the information is absent but because it is not reliably attended to. Architecturally, an overly broad CONSIDERING clause couples the reasoner tightly to the ENTITY's full schema: any change to the ENTITY's field list requires review of the reasoner's behavior, because the additional fields may shift the model's attention distribution in unpredictable ways. A precisely specified CONSIDERING clause — projecting only the fields the reasoner genuinely requires — is both more robust to schema evolution and more reliably attended to by the model. The discipline of designing a minimal, semantically complete CONSIDERING clause is one of the distinguishing marks of experienced REASONER design.

The **system prompt** is the component of the REASONER that does not appear in the `.agi` declaration syntax but is nevertheless its most operationally significant element. The USING, CONSIDERING, PRODUCING, and MODEL clauses specify the structural scaffold of the inference run — what data, what fields, what output type, what model. The system prompt specifies what to do: the instructions, the quality criteria, the tone, the format, the reasoning approach, the error handling. The system prompt is where the practitioner's domain expertise is encoded. A REASONER declaration without a carefully designed system prompt is a scaffold with no building on it. It will execute — it will assemble records, pass them to the model, and produce a string output — but the output will reflect the model's defaults rather than the practitioner's specification. Agicore associates each REASONER declaration with a system prompt file referenced by name in the declaration or embedded as a string field; the compiler validates the reference but cannot validate the prompt's semantic quality. The responsibility for system prompt quality rests entirely with the practitioner. This is the asymmetry in Agicore's compile-time guarantees: structural correctness is enforced automatically; semantic quality is the practitioner's obligation. The practitioner who treats the system prompt as an afterthought — filling it in last, revising it rarely, sharing it without documentation — is accepting the same quality risk as the engineer who ships untested code. The system prompt is the specification. It deserves the same rigor as any other engineering artifact in the `.agi` file.
