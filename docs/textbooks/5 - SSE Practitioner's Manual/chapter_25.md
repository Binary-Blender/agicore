# Chapter 25: Inline QC Contracts and the Shadow Line

The previous chapter described the NBVE as a single engine evaluating Primary and Variant paths through a pipeline. That description is correct at the architectural altitude. It is not enough at the implementation altitude. Two specific lower-level components have to exist before the engine has anything to do, and most practitioners who try to build the engine without these components in place produce something that looks like an NBVE and behaves like an expensive logging system. This chapter develops those components. Inline QC contracts at every node. A shadow line that runs variants without disrupting production. Both are operational primitives the practitioner builds into the pipeline from day one. Both have specific shapes. Both have specific failure modes when built wrong.

Start with the QC contract, because the contract is the precondition for everything else. A QC contract is a typed statement of what a node's input must look like and what its output must look like, attached to the node itself so that the contract is checked every time the node runs, in every environment, against every input. The contract is not a comment. The contract is not documentation. The contract is executable, and the execution happens inline with the node's operation. If the contract is violated — input does not match the declared input shape, output does not match the declared output shape — the violation is recorded and the pipeline either halts, retries, or routes to a recovery path, depending on how the contract is configured to behave on violation. The contract is the unit of evaluation that the NBVE's critics aggregate over.

Why inline. The word matters and I want to be specific about it. A contract that lives in a separate test suite that runs occasionally is not an inline contract. A contract that runs only against the final pipeline output is not an inline contract. An inline contract runs at every node of the pipeline, every time, as part of the node's normal operation, with the node's output not propagating downstream until the contract is satisfied or the violation is handled. The inlining is what gives the contract its operational meaning. A contract that runs occasionally catches violations occasionally. A contract that runs inline catches every violation, and the violations either get handled before they propagate or get recorded as failures the engine can aggregate.

The contracts have a specific shape. Inputs are typed. Outputs are typed. The types are not just JSON shapes — JSON-shape checking is the easy part — but include semantic constraints that JSON shape cannot express. The input to a summarization node is not just "string"; it is "non-empty string under N tokens, in language L, that the upstream node has flagged as containing the kind of content this summarizer is configured to handle." The output is not just "string"; it is "non-empty string under M tokens, in language L, whose first paragraph contains a topic sentence matching the input's subject, whose tone matches the baseline document the Tone Anchor references." The semantic part of the contract is what the critics from the previous chapter operationalize. The Spec Guard checks the structural part. The Logic Police checks the consistency of the semantic part with the input. The Tone Anchor checks the calibration of the semantic part against the baseline. Each critic is a partial executor of the contract. The contract is the spec the critics implement against.

The contract-first discipline. This is the part of the chapter that is uncomfortable to write because it requires telling practitioners to do something they will not want to do. Before you build the pipeline, you write the contracts. Not after. Not in parallel. Before. The practitioner who writes the pipeline first and the contracts second produces pipelines whose contracts are post-hoc rationalizations of whatever the pipeline happened to be doing — contracts that ratify the existing behavior rather than constraining it toward a target behavior. The post-hoc contract has no operational value because the pipeline already satisfies it by construction. The contract that constrains is the contract that came first, that the pipeline was built to satisfy, that the pipeline can fail to satisfy when something drifts. The discipline is to write the contract before the code. The discomfort is real. The discipline is non-negotiable.

I will be specific about what writing a contract looks like in practice, because the abstract version of contract-first development tends to be vague enough that practitioners can convince themselves they are doing it while not doing it. Writing a contract for a node means producing a short document — a markdown file, a YAML stanza, a typed schema in whatever language the pipeline is written in — that names the node, declares the input shape with both structural and semantic constraints, declares the output shape with both structural and semantic constraints, and declares the violation behavior. The document goes into source control. The document is referenced by the node's implementation. The document is the source of truth for what the node is supposed to do. The implementation is the source of truth for what the node actually does. The two are checked against each other every time the node runs. The discipline is not difficult to describe. The discipline is difficult to actually maintain across the months that the pipeline is being built and modified. The maintenance is what makes the difference.

The shadow line. Now the second component. The shadow line is the parallel execution infrastructure that runs Variant Paths concurrently with Primary Paths without their outputs interfering. The shadow line is the runtime mechanism that makes the NBVE possible. The contract is what gets checked; the shadow line is the substrate on which the checking happens for the variant alongside the primary.

The shadow line has a specific operational property that is worth naming because it is the property practitioners get wrong most often. The shadow line is invisible from downstream. The variant runs. The variant's output gets evaluated. The evaluation gets logged. The variant's output never enters the downstream system. Never. Not partially. Not in test mode. Not with a feature flag that some downstream consumer might toggle. The variant's output is contained inside the engine. The contract evaluation of the variant produces a record. The record is what propagates. The output does not. This is the property that distinguishes the shadow line from production traffic splitting. Production traffic splitting routes some real traffic to the variant; the shadow line routes no real traffic to the variant. The variant runs on the same inputs the primary ran on, in parallel, with its outputs collected and not consumed. The shadow line is dark. The darkness is the safety property.

The implementation pattern, because this chapter exists to be operational. In its simplest form, the shadow line is a wrapper around the node call. The wrapper takes the input, calls the primary version of the node, calls the variant version of the node (if a variant is currently configured for this node), captures both outputs, evaluates both against the node's contract via the critics, logs the comparison record, and returns only the primary output to the caller. The caller has no idea a variant ran. The wrapper is the integration point. The wrapper exists at every node where shadow evaluation is enabled. The wrapper is what the Agicore `evaluateWithShadow` helper from Vol II Chapter 18 is the runtime-emitted version of. The practitioner pipeline form is the same shape — wrapper, primary, variant, contract check, log record, primary returned — implemented in whatever language the pipeline happens to be written in.

The data collection discipline. This is the part of the chapter where I have to talk about what is being collected and why, because the data is what makes the engine more than a logging system. Every comparison the shadow line produces is a record. Each record contains: the node id, the timestamp, the input, the primary output, the variant output, the variant's configuration delta from primary, the contract-check result for each output (which critic flagged what), the token costs and latencies of each path, and any downstream propagation data the pipeline structure supports. The records accumulate in a structured store — a database, a log file with a strict schema, a parquet directory; the substrate matters less than the structure. The records are not write-and-forget. The records are the dataset the next chapter operates against. The shadow line's value is exactly proportional to the discipline with which its records are captured, preserved, and made queryable.

The discipline tends to fail in two specific ways. First, the records get captured but never queried. The shadow line runs, the data accumulates, and the practitioner never goes back to look at the data because the data has no place in the working week's calendar. This is the silent failure mode. The infrastructure exists; the loop does not close. The records become residue. The records that are not queried are records that are not informing decisions, and records that are not informing decisions are records that have produced no value. The second failure mode is the opposite. The records get queried but never make it into the contract revisions or the prompt template updates or the pipeline architecture changes the data was supposed to inform. The data sits in a notebook somewhere. The dashboard exists. The practitioner reads the dashboard once a week and feels informed and changes nothing. This is the false-feedback failure. The records have informed the practitioner's mental model; the records have not informed the system. The system has not changed.

Both failure modes are real. Both failure modes are common. The defense against both is the same — the records get reviewed on a calendar that drives explicit decisions, and the decisions get logged. Friday morning review of the week's shadow-line records. The review produces a list of specific actions — this contract needs to be tightened, this prompt template needs revision, this variant is ready to be promoted, this variant should be retired. The actions go into the next week's working calendar. The next Friday's review checks whether the actions were taken and what their downstream telemetry shows. The loop closes. Without the closing loop, the shadow line is decoration. With it, the shadow line is the substrate that all of the chapter 26 tuning operates on.

The deeper shift the architecture supports. I want to name this clearly because the architectural detail can obscure the conceptual change, and the conceptual change is the part that actually matters for how the practitioner thinks about the work.

Pre-NBVE, the question a practitioner asks about a pipeline is "did this output work?" The question is asked retrospectively, against finished outputs, by a human reviewer whose attention is the bottleneck. The question is a deployment question. It is asked after the pipeline has run. It is asked about a specific output, not about the system that produced it. The answer feeds, at best, into a future revision of the pipeline configuration. The lag between the failure event and the configuration update is days or weeks. The practitioner is running the pipeline blind during the lag.

Post-NBVE, the question shifts. The question becomes "how should we tune the production system right now?" The question is asked in real time, against aggregate telemetry, by an engine whose attention does not degrade. The question is a tuning question, not a deployment question. It is asked while the pipeline is running. It is asked about the system, not about a specific output. The answer feeds directly into the next variant configuration the shadow line evaluates. The lag between observation and adjustment is the duration of the next shadow-line window — hours or days, not weeks. The practitioner is no longer running blind. The practitioner is running with telemetry that closes the loop fast enough to be operational.

This shift is the difference between testing a release candidate and tuning a manufacturing system. The release-candidate framing treats outputs as the deliverable and the pipeline as the static substrate that produces them. The manufacturing-system framing treats the pipeline as the deliverable and the outputs as samples of the deliverable's behavior. The release-candidate framing improves the deliverable. The manufacturing-system framing improves the substrate. The manufacturing-system framing is what produces the leverage that compounds, because every improvement to the substrate is an improvement to every future output the substrate produces. The release-candidate framing produces improvements that apply to one release and have to be re-discovered for the next one. The two framings produce visibly different working practices, and the difference compounds over months. The practitioner who has internalized the manufacturing-system framing is the practitioner whose pipelines get measurably better quarter over quarter. The practitioner who is still in release-candidate framing is the practitioner whose pipelines stay roughly the same, with the variance moving around inside fixed bounds.

A note on the relationship between the two components, because the chapter has developed them separately and the relationship is worth naming explicitly. Inline QC contracts are the static specification of what the pipeline is supposed to do at each node. The shadow line is the dynamic execution mechanism that produces evidence about whether the pipeline is doing what it is supposed to. The contracts without the shadow line produce a pipeline whose specification is checked against single executions, which catches structural violations but does not produce the comparative telemetry the engine from Chapter 24 operates against. The shadow line without contracts produces an execution mechanism whose output is comparison records the practitioner cannot interpret, because there is no specification against which to interpret them. Each component is necessary for the other to be operationally meaningful. The two together produce the substrate that supports tuning. Either alone produces something that looks like the substrate but is not.

Another note on what the contracts surface, because contracts that are well-written do something most practitioners do not expect. They surface ambiguity in the practitioner's own understanding of what the pipeline is supposed to do. The practitioner sits down to write the contract for a node and discovers, partway through, that the node's job is not actually well-specified — that the practitioner had been operating on a fuzzy sense of what the node should produce, that the fuzziness had been hidden by the node mostly producing acceptable output most of the time, that the fuzziness had to be resolved before the contract could be written. The resolution is part of the value. The act of writing the contract forces specification at a level of precision that the practitioner had been comfortable evading. The precision is what makes the contract evaluable. The precision is also what makes the practitioner's thinking about the pipeline sharper. The discipline of contract-writing improves the practitioner as a side effect of producing the contracts. The improvement is real and persistent and outlasts any specific pipeline the practitioner happens to be working on at the time.

A third note, on the question of how strict the contracts should be. There is a calibration question here that practitioners ask early and that has no general answer. Contracts that are too strict reject outputs that would have been operationally acceptable and produce more retries than the pipeline needs. Contracts that are too loose accept outputs that should have been caught and let degradation propagate downstream. The right calibration is pipeline-specific and discovered empirically. The starting point is to write contracts at the level of strictness the practitioner believes the pipeline requires, run them for a week, look at the violations the contracts produced, and adjust. Violations that surfaced outputs the practitioner would have rejected on manual review are correctly flagging — keep the contract strict. Violations that surfaced outputs the practitioner would have accepted are over-flagging — loosen the contract. The calibration converges over a few cycles. The converged contract is the contract that flags exactly the failures the practitioner cares about and nothing else. The converged contract is rare in newly-built pipelines and standard in pipelines that have run the discipline for a quarter. The convergence is a deliverable. It is not free.

How to actually build this from where you currently are. The chapter has to end with the move. I will be specific. Pick a pipeline you run that has at least one stable Primary Path. Write the contract for one node — the node you are most suspicious of, the node whose outputs you find yourself checking most often, the node whose failures cost you the most when they happen. The contract is a markdown file with the structural and semantic constraints declared. Wire the contract into the node's implementation so that the contract is checked every time the node runs. Log the contract-check results to a file. Run the pipeline for a week with the contract in place. Look at the log file at the end of the week. The log file will tell you things about the node's behavior that you did not know — the contract is the diagnostic instrument, not just the constraint. The first contract takes a working day. The first week of log data takes a working week. After two cycles of contract-then-log-review, the discipline starts producing decisions that change the pipeline in ways that show up as fewer downstream surprises. After a quarter of the discipline, you have a pipeline with contracts at every node, a log substrate that captures their evaluation, and a working week structure that includes the review of the log substrate. That is the state from which Chapter 26 — the tuning chapter — actually becomes possible. The shadow line layer follows the contract layer; you do not build the shadow line without contracts in place, because the shadow line has nothing to evaluate against.

**The Bottom Line:** Write contracts before you write nodes. Run the shadow line dark, with its records collected on a schedule you actually review. The shift is from "did this output work?" to "how should we tune the production system right now?" The first question is a deployment question. The second question is the question that runs a manufacturing system. The architecture in this chapter is what lets you ask the second question.
