# Chapter 26: Tuning the Manufacturing System

The previous two chapters built the engine and the substrate it runs on. This chapter is about what you do with the telemetry the substrate produces. The chapter is the most operational in the book because the practice it describes is the practice that turns the engine from a piece of architecture into a working productivity multiplier. The engine produces data. The data goes nowhere unless it produces decisions. The decisions are the four kinds I will name in this chapter, and the way the decisions chain together is what makes the difference between a pipeline that ages well and a pipeline that quietly degrades into the LLM regional mean.

The shadow line accumulates records. Each record is a comparison — primary output, variant output, contract evaluation by the critics, token cost, latency, downstream telemetry. After a week of running, the record store contains hundreds of records per pipeline node. After a month, thousands. After a quarter, enough records to support real statistical signal. The records exist to support exactly four kinds of tuning decisions, and I want to walk each of them in operational detail.

The first dimension is model selection. The pipeline runs against a specific model. The model is one of the most expensive components of the pipeline by cost-per-output. Most pipelines, when first built, default to running everything on the most capable model the practitioner has access to, because the most capable model is the safest choice during the build-up phase when failure modes are not yet characterized. The model is overpowered for most of the work the pipeline does. The pipeline could, in principle, run on a model an order of magnitude cheaper without quality degradation. The question is which steps of the pipeline can absorb the cheaper model and which steps cannot. The NBVE answers this question empirically. Primary Path runs the expensive model. Variant Path runs the cheaper model. Contract evaluation runs on both. The records accumulate over a window long enough to produce statistical signal — usually a thousand comparisons or so for a stable-volume pipeline, which is days or weeks depending on traffic. The aggregate of the records tells you whether the cheaper model satisfies the contract at a comparable rate to the expensive model. If it does, the cheaper model gets promoted to Primary Path, the expensive model retires from that node, the pipeline's cost per output drops by the order of magnitude the model differential represents. If it doesn't, the cheaper model retires from the variant slot, the practitioner has learned something specific about which kind of work that node does that requires the expensive model's capabilities, and the next variant experiment can be aimed at a different cost reduction. Either outcome is productive. Both produce decisions that improve the pipeline's economics. The records are the substrate against which the decisions are made.

The second dimension is prompt engineering. The pipeline's prompts are the most accessible component to modify and the component that, in most pipelines, has the highest expected return per hour of practitioner attention. The default prompt structures for a node are usually first-draft prompts that were good enough to ship the pipeline and have not been revisited since. The first-draft prompts are almost never optimal. They contain instructions that are not load-bearing, they contain phrasings that route to suboptimal regions of the model's latent space, they fail to leverage the practitioner signaling techniques developed in earlier chapters, they leave constraints implicit that should be explicit. The shadow line evaluates revised prompt structures. Variant Path runs the revised prompt; Primary Path runs the baseline. The contract evaluation captures whether the revised prompt produces outputs that satisfy the contract at a higher rate than the baseline does, or whether it produces outputs that are roughly comparable but cheaper, or whether it produces outputs that are worse in ways that the contract surfaces. The records give the answer empirically. The practitioner writes a revised prompt, runs it against shadow for a window, looks at the comparison. The revised prompt either earns promotion or doesn't. The next revised prompt builds on whatever the data showed about the previous revision. The discipline is the discipline of any iterative engineering process — change one variable at a time, run long enough to see signal, decide based on signal rather than intuition, repeat. The discipline is unremarkable as an engineering discipline. The discipline is unusual in AI work because most practitioners are still operating under the release-candidate framing from the previous chapter and have not built the substrate that supports the discipline. With the substrate in place, the discipline is simply what tuning looks like.

The third dimension is deterministic tuning. By deterministic tuning I mean the configuration parameters of the underlying model invocation that are not the prompt itself — temperature, top-p, max tokens, retry strategy, structured output enforcement, response format constraints. These parameters are the most overlooked component of the pipeline because their effects are subtle and their defaults usually work well enough. The defaults are not optimal. A summarization node that runs at temperature 0.7 because that was the default is producing outputs with more variance than the contract requires, which means more contract violations and more retries than the same node would produce at temperature 0.3. A generation node that runs at temperature 0.3 because the practitioner read once that lower temperature is safer is producing outputs with less creative range than the contract would tolerate, which means missed opportunities for outputs that satisfy the contract while being measurably more interesting. The shadow line evaluates temperature variants. Variant Path runs at a different temperature; Primary Path runs at the baseline temperature; the contract evaluation captures whether the temperature shift improved or degraded the contract pass rate. The records support the decision. The same approach applies to top-p, to retry strategies, to structured-output enforcement levels, to all of the deterministic parameters that surround the prompt itself. Each parameter is a candidate variable for a shadow-line experiment. Each parameter, tuned correctly against the contract, contributes a small improvement. The small improvements compound across the pipeline's nodes. The compounded improvements, over a quarter of consistent tuning, produce pipelines that operate visibly differently from pipelines that have not been tuned. The differences are not subtle.

The fourth dimension is sub-pipeline resilience. This is the operational layer that handles the failure cases — the retries, the recovery loops, the fallback paths the pipeline takes when a node's first attempt fails its contract. Resilience is the dimension most practitioners under-engineer because the failure cases are rarer than the success cases and the practitioner's attention naturally focuses on the success path. The under-engineering is expensive. A pipeline whose retry strategy is "if the node failed, run it again with the same configuration" produces a particular pattern of cascading failures — the same input produces the same kind of failure twice, the retry is wasted, the pipeline either fails for the input or escalates to a more expensive fallback. A pipeline whose retry strategy is "if the node failed, identify which critic flagged the failure and run a revised configuration designed to address that specific failure mode" produces dramatically better recovery, because the second attempt is informed by what went wrong in the first. The shadow line evaluates recovery strategies. Variant Path runs a candidate recovery loop against the failure cases; Primary Path runs the baseline. The records capture the recovery rates. The candidate either earns promotion as the new baseline recovery strategy or doesn't. The discipline compounds. Over months of running, the pipeline accumulates a library of failure-mode-specific recovery strategies, each one validated against shadow-line evidence, and the pipeline's effective reliability — the rate at which it produces contract-satisfying output on the first or second attempt — climbs from whatever the baseline was toward an asymptote that is meaningfully above what an unoptimized pipeline produces.

Four dimensions. Model selection, prompt engineering, deterministic tuning, sub-pipeline resilience. Each dimension is a class of experiments the shadow line can support. Each experiment is a variant that runs against a primary, with the contract critics evaluating both, with the records accumulating into a dataset that supports the promotion decision. The discipline at any given time is a portfolio of experiments across the four dimensions, each at a different stage of its evaluation window, each producing telemetry that informs the next experiment. The practitioner's working week, post-NBVE, is structured around this portfolio. Friday-morning review of the records. Decisions about which experiments are ready for promotion, which need more data, which should be retired, which new experiments to spin up. The decisions go into the calendar. The calendar drives the next week of shadow-line work. The loop closes weekly.

I want to name a distinction that the chapter has been circling and that deserves to be named precisely. The distinction is between process research and competitive selection, and the distinction matters because most practitioners trying to use AI evaluation are doing competitive selection while believing they are doing process research, and the conflation produces worse pipelines, not better ones.

Competitive selection is what happens when two model versions or two prompt structures compete against each other on a benchmark, and the winner is selected for production use. The practice is unobjectionable when used appropriately. The practice is also limited. Competitive selection produces a binary outcome — A or B — and the binary outcome carries very little signal about why one option performed better than the other. The reasons are aggregated into the benchmark score. The aggregation collapses the information. The next round of competitive selection has to rediscover whatever the previous round implicitly learned, because the previous round's learning was compressed into a single number.

Process research is what NBVE supports natively. Process research is not asking which option wins; process research is asking how the system behaves and what specifically the system is sensitive to. The shadow line's records are full of the data process research operates on — which kinds of inputs cause contract violations, which kinds of contract violations correlate with which configuration variables, which critics catch which classes of failure, what the latency-cost-quality tradeoff curve actually looks like for this specific pipeline as the variant configuration sweeps across its space. Process research produces understanding of the manufacturing system. The understanding is what informs the next design decision, the next contract revision, the next architectural change. The understanding compounds across decisions in a way that competitive selection's binary outcomes cannot. Process research is the framing that makes the NBVE the substrate for a continuously improving pipeline rather than a substrate for repeated bake-offs.

The shift in framing matters because practitioners coming to NBVE from a software-engineering background tend to import the A/B testing mental model, which is competitive-selection-shaped. The A/B testing mental model is wrong for this work. A/B testing assumes the variants are roughly fixed and the question is which one performs better against a chosen metric. NBVE supports a different practice — the variants are not fixed, they are vehicles for asking specific questions about the system's behavior, and the records the shadow line produces are the data the practitioner uses to refine the questions. The questions get sharper as the practitioner spends more time with the data. The pipeline gets better not because the practitioner ran more bake-offs but because the practitioner understood the manufacturing system more deeply, and the deeper understanding informed better design decisions across all four dimensions.

A worked example, because the chapter is supposed to be operational. A pipeline I run produces structured chapter expansions from rough outlines for several different book projects. The pipeline ran for months with a single model on a single prompt template, and produced acceptable output most of the time. Acceptable was the baseline. The shadow line started running variants. The first month's variants were prompt-level — small revisions to the prompt structure, testing different framings of the chapter expansion task. Half the revisions improved contract pass rates; half degraded them. The data from the degraded revisions taught me which constraints in the prompt were load-bearing and which were redundant. The successful revisions got promoted, and the pipeline's pass rate on first attempt climbed from 70 percent to about 84 percent over the month.

The second month I ran model-selection variants against the prompt-tuned baseline. Two cheaper models tested. One produced outputs that passed the contract at a rate within two percentage points of the expensive baseline, at one-eighth the cost. That model got promoted to primary. The expensive model now runs only on the fallback path, when the cheaper model produces output that fails the contract. The cost per chapter expansion dropped by something like 75 percent. The quality, by every measurable contract-evaluation metric, stayed within statistical noise of the baseline. Neither outcome would have been visible without the shadow-line records to support the decision.

The third month I ran deterministic-tuning variants — temperature, max tokens, structured output constraints. The temperature variant turned out to be the most important; the baseline temperature had been left at the platform default, which produced more stylistic variance than the contract required. Lowering the temperature reduced the variance and improved the contract pass rate by another six points. The fourth month I ran resilience variants — the recovery strategy when the cheaper model produced contract-failing output. The variant strategy was to run the recovery with a prompt that explicitly pointed at the specific contract violation, rather than retrying with the same configuration. The variant strategy produced contract-passing recovery output about 85 percent of the time, against the baseline's 50 percent. Recovery loops got rewritten across the pipeline.

Four months of disciplined shadow-line tuning produced a pipeline whose cost per output was about 70 percent lower than the starting baseline, whose contract-pass rate on first attempt was about 90 percent against the starting 70 percent, and whose recovery rate on second attempt was about 85 percent against a baseline of around 50 percent. The pipeline did not change in any architectural way during the four months. The pipeline got tuned. The tuning was empirical, was grounded in the shadow-line records, and proceeded one experiment at a time across the four dimensions. The compounding was small per experiment and large across the portfolio. Nothing in the practice was difficult. The practice was, simply, the practice. The substrate from the previous chapter supported it. The framing from this chapter directed it. The output was visibly different at the end of the quarter than at the start.

A note on the order of operations across the four dimensions, because practitioners new to the discipline often want a recipe and there is, approximately, one. Prompt engineering first, because the prompt is the highest-leverage variable and the cheapest to vary. Model selection second, because the right model is only knowable once the prompt is tuned — testing a cheaper model against an unoptimized prompt rejects the cheaper model for failures that better prompting would have eliminated. Deterministic tuning third, because temperature and structured-output settings interact with both the prompt and the model and have to be tuned against the combination rather than against either alone. Resilience fourth, because resilience strategies depend on understanding what kinds of failures the pipeline produces, and that understanding requires the prior three dimensions to have stabilized enough that the residual failures are characterizable. The order is not a hard rule. The order is the default sequence I have seen produce the fastest convergence in pipelines I have built and watched others build. A practitioner who starts with model selection before tuning the prompt usually re-does the model selection later, because the data from the first selection becomes obsolete when the prompt changes. The order saves rework.

Another note on how the four dimensions interact, because the dimensions are not orthogonal and treating them as orthogonal produces tuning loops that thrash. A prompt revision that improves contract pass rates at the current temperature setting may degrade them at a different temperature setting; the prompt and the temperature interact in the model's response. A model change that improves pass rates against the current prompt may not produce the same improvement against a revised prompt; the model and the prompt interact. The interactions mean that the four dimensions need to be tuned in coordinated cycles rather than in disconnected experiments. The discipline is to tune one variable at a time within a cycle, but to revisit prior decisions when adjacent variables have shifted enough that the prior decisions may no longer be optimal. The revisiting is a property of mature tuning practice. A practitioner who promotes a configuration in week three and never revisits it in week ten is a practitioner who is leaving leverage on the table, because the configuration that was optimal in week three may not be optimal against the configuration the pipeline has evolved into by week ten. The revisits are cheap. The shadow line still runs. The revisit experiment is a small variant against the current baseline. The discipline includes scheduled revisits of previously-promoted decisions, on a cadence that the pipeline's volatility supports.

A third note, on what to do when the data is ambiguous. Some shadow-line experiments produce clear signal — the variant pass rate is meaningfully higher or lower than the primary, the cost differential is large, the decision is obvious. Some experiments produce noise — the pass rates are within statistical error of each other, the cost differential is small, no clean decision is supported. The temptation is to make a decision anyway, usually based on the practitioner's prior intuition about which option should be better. The temptation should be resisted. The right response to ambiguous data is to either run the experiment longer until the data is unambiguous, or to retire the experiment and recognize that the variation tested does not matter at the resolution the pipeline operates at. Forcing a decision on ambiguous data introduces noise into the pipeline's configuration; the pipeline drifts in directions the data does not support, and the drift accumulates across multiple ambiguous-decision events into a pipeline whose configuration is not justifiable by any specific evidence. The discipline of holding decisions until the data supports them is what keeps the tuning grounded. The grounding is what produces the compounding. The compounding is the point.

What this looks like over longer timeframes. The four-month example is one quarter. Run the discipline for a year. The cumulative tuning across four quarters of consistent practice produces pipelines whose performance envelope is unrecognizable from the starting envelope. Run it for three years. The pipelines you operate are operating at characteristics that practitioners who have not run the discipline cannot match without running the same multi-year process. The leverage from Chapter 22 is the leverage of the cumulative cognitive infrastructure. The leverage from this chapter is the leverage that the cumulative infrastructure produces against the manufacturing system. The two compounding curves are different. The two curves overlap into something that has no clean name. The closest name I have for it is what mature practice looks like.

**The Bottom Line:** You're not picking model versions or testing prompts. You're tuning a manufacturing system of intelligence across four dimensions — model, prompt, deterministic parameters, resilience — using the shadow line as the substrate that makes the tuning empirical. Process research, not competitive selection. Run it continuously. The compounding shows up in months.
