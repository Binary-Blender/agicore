# Chapter 7: Code Generation Theory

The reflexive way to talk about code generation is to call it templating. Templates are easy to picture: a string with holes in it, a context with values in it, an interpolation step that pours one into the other. Most code-generation systems people have encountered — the report writers that emit HTML, the scaffolding tools that emit a directory of starter files, the IDE wizards that emit a class skeleton — operate on this model. Templating is a perfectly reasonable engineering technique for problems where the output is independent. Reports do not have to type-check against each other. Scaffolds are starting points the developer immediately edits. The output is consumed by a human, who can resolve any inconsistencies the template missed.

Agicore code generation is not templating, in any operationally important sense, and the difference matters because it is the difference between a tool that emits convenient starter files and a system whose emitted artifacts are themselves a load-bearing layer of the discipline. Each Agicore generator is a typed function from the validated AST to one or more output files. The generators run as a group, share a common derived model of the program, and emit artifacts that must remain coherent with each other under two independent downstream type checkers. The output of the Rust generator is consumed by the TypeScript generator's expectations and vice versa, not directly but through the type contracts that both encode from the same AST. A bug in any one generator that breaks the coherence will be caught by `cargo build` or `tsc --noEmit`. The generators are not free to emit whatever feels right; they are pinned by a cross-generator coherence contract that is enforced mechanically.

This chapter develops the discipline. The argument runs in four moves. First, generators as functions rather than templates — what shifts when you treat them as typed code rather than as string interpolation. Second, the cross-generator coherence contract — what it means for the Rust generator and the TypeScript generator to produce coherent outputs, why the AST is the only shared source they can use, and how the coherence is enforced. Third, stable output as a discipline — why byte-for-byte reproducibility of generated code is worth the effort it takes to maintain, and what techniques produce it. Fourth, the protected-file contract — how the compiler reserves a small, well-marked region of every generated project for human implementation, and why that region is the right shape rather than some other shape.

A generator, in Agicore, is a TypeScript function whose signature reads roughly:

```typescript
function generateRustCommands(program: Program): GeneratedFile[]
function generateTypeScriptInvokes(program: Program): GeneratedFile[]
function generateSqlMigration(program: Program): GeneratedFile[]
function generateReactComponents(program: Program): GeneratedFile[]
function generateTauriConfig(program: Program): GeneratedFile[]
```

The input is the AST. The output is a list of files, each carrying a path and a string of contents. The function is pure: same AST, same output, no side effects, no mutable state outside the call. The purity is not aesthetic; it is what makes the generators testable. The compiler's test suite — currently more than three thousand passing tests — is dominated by tests of the form *given this AST fragment, the generator should produce this output*. Tests of pure functions of typed data are short, fast, and unambiguous. The compiler's authors can change a generator confidently because the test suite will catch any output that breaks the recorded contract.

Generators-as-functions has a second consequence beyond testability: it makes the cross-generator contract expressible. The Rust generator emits a function with a particular signature. The TypeScript generator emits an invoke wrapper that must match that signature. Both generators consume the same AST node. A change to the action's `INPUT` declaration in the AST changes the Rust function's parameter list, and the TypeScript wrapper's parameter list, in lockstep. There is no risk that the developer added a parameter on one side and forgot to add it on the other, because no developer added it on either side. The generator did, from the same AST node, for both languages. The coherence is enforced by shared derivation rather than by shared discipline.

To make the point concrete, return to the running example and watch the same `ENTITY Student { name: string }` produce coherent output across three languages. The SQL generator emits:

```sql
CREATE TABLE IF NOT EXISTS students (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL
);
```

The Rust generator emits, in `src-tauri/src/commands/crud.rs`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Student {
    pub id: i64,
    pub name: String,
}

#[tauri::command]
pub fn list_students(state: tauri::State<DbPool>) -> Result<Vec<Student>, String> {
    let conn = state.0.lock().unwrap();
    let mut stmt = conn.prepare("SELECT id, name FROM students").map_err(|e| e.to_string())?;
    let rows = stmt.query_map([], |row| {
        Ok(Student { id: row.get(0)?, name: row.get(1)? })
    }).map_err(|e| e.to_string())?;
    rows.collect::<Result<Vec<_>, _>>().map_err(|e| e.to_string())
}
```

The TypeScript generator emits, in `src/lib/types.ts` and `src/lib/invokes.ts`:

```typescript
export interface Student {
  id: number;
  name: string;
}

export async function listStudents(): Promise<Student[]> {
  return invoke<Student[]>('list_students', {});
}
```

Three artifacts, three languages, one source of truth. The table name `students` and the field name `name` appear in the SQL. The struct field `name` appears in the Rust struct with the type `String`. The interface field `name` appears in the TypeScript with the type `string`. The command name `list_students` appears as the snake_case name in the Rust `#[tauri::command]` registration and as the snake_case string literal in the TypeScript `invoke` call. The TypeScript function is `listStudents` — camelCased, because TypeScript convention is camelCase. The case conversion happens in exactly one place, in a small utility consumed by every generator, and the rule is one-directional: snake_case on the database and IPC side, camelCase on the JavaScript side. The developer never has to remember which side they are on, because the generator made the conversion for them.

This is the cross-generator coherence contract in action. Three generators ran independently, consumed the same AST node, and produced artifacts that are type-coherent. If the developer changes the field from `name: string` to `full_name: string` in the `.agi` source, the next regeneration produces SQL with a `full_name` column, a Rust struct with a `full_name` field, a TypeScript interface with a `fullName` field, an invoke wrapper that destructures `fullName`. The change propagates everywhere by the same mechanism that produced the initial coherence. The developer touched one line in one file. The generated artifacts moved together.

Compare this to the reflection-based alternative, where coherence is enforced at runtime by inspecting types and dispatching dynamically. Reflection moves complexity to runtime, where errors are invisible until exercised. It is also fundamentally hostile to AI-assisted authoring, because the AI has no way to verify the cross-component coherence of a reflective design without running the system. Generation is the inverse trade. Complexity moves to build time, where errors are visible immediately. AI authors generate against a parser-and-validator pair that produces yes-or-no answers in milliseconds. The cost of generation is that the generators themselves must be written, maintained, and tested. The cost of reflection is that every runtime invocation pays the cost of dynamic dispatch and every bug surfaces in production. The two approaches optimize for different things; the compiler-first architecture necessarily favors generation.

The third move is about output stability. Same input, same output, byte-for-byte, forever. This sounds like a property that should come for free from pure functions, and at one level it does — a pure function of the same input is mathematically the same function call and produces the same value. But the value, in this case, is a string with a specific layout, and several small implementation choices can make the same logical output render as different bytes across runs. The compiler authors have been careful about every one of them. Hashmap iteration is sorted before being emitted, because hashmaps in many languages do not guarantee iteration order across runs. Sibling outputs are sorted by a deterministic key, typically the declaration name. Timestamps are never embedded in generated files, because timestamps make every regeneration look like a code change in the version-control diff. Generated marker comments are precisely the strings `// @agicore-generated` and `// @agicore-protected`, with no version number or environment information that might vary across machines.

The payoff for this discipline is large and not immediately obvious. Generated code lives in the repository. It is committed, reviewed in pull requests, audited in security reviews, blamed when bugs are tracked. If regenerating an `.agi` file produces a diff that includes a hundred lines of cosmetic reshuffling alongside the three lines of real change, the reviewer cannot tell the difference and the audit trail is destroyed. If regeneration produces a diff that includes exactly the three lines that correspond to the source change, the reviewer can verify the change in seconds and the audit trail is intact. Stable output is what makes the generated code reviewable. Reviewable generated code is what makes the architecture trustworthy to operate at scale. The discipline is not free, but the alternative — a system where generated diffs are noisy enough to discourage review — is the slow death of the audit property.

One operational consequence of the stable-output discipline is worth surfacing because it affects how the team works day-to-day. Continuous integration pipelines for Agicore projects routinely run the compiler against the committed `.agi` source and diff the result against the committed generated code. If the diff is non-empty, CI fails. The check is the mechanism by which the project guarantees that the committed generated code is genuinely the output of the committed source — that no one has hand-edited a generated file, that no one has forgotten to regenerate after changing the source, that no one has merged a branch where the source and the generated output drifted apart. The check costs seconds to run and catches a category of drift that would otherwise accumulate silently. Without the stable-output discipline the check would be unworkable, because the diff would be dominated by cosmetic noise. With it, every diff is genuine. The discipline pays for itself the first time it catches a drifted commit.

The fourth move is about the protected-file contract. Most of the generated project is recreated wholesale on every regeneration; this is the right default, because it means the generated state is always exactly what the current AST implies. But there is a category of file where wholesale regeneration would be wrong: the implementation body of an `ACTION IMPL` declaration. When the DSL says `ACTION fetch_weather_data { IMPL "fetch_weather_data_impl" INPUT location: string OUTPUT temp: float }`, the developer's job is to write the Rust function body that actually calls the weather API and parses the response. The compiler's job is to emit a Rust file with the correct signature, the correct module wiring, and the right Tauri command registration — and then leave the function body alone, forever, until the developer changes the DSL in a way that demands a signature change.

The mechanism is a single marker. The first line of the generated stub file is `// @agicore-protected`. When the compiler is about to emit such a file, it first checks whether a file at that destination path exists and begins with that marker. If it does, the compiler leaves it untouched. If it does not, the compiler emits the stub. The stub looks roughly like this:

```rust
// @agicore-protected
// This file is generated once and protected from regeneration.
// Edit the function body. The compiler will keep dependent wiring in sync.

use crate::commands::types::*;

#[tauri::command]
pub async fn fetch_weather_data(location: String) -> Result<FetchWeatherDataOutput, String> {
    // TODO: implement
    Err("not implemented".to_string())
}
```

The developer fills in the body. On the next regeneration, the file is untouched. Other files — the command registration list, the TypeScript invoke wrapper, the store slice that calls the wrapper — are regenerated freely, because they reference the function by its signature and the signature comes from the DSL. The protected file holds only the implementation. The contract is precise: you fill in the body; we keep the wiring around your body in sync. The signature in the protected file matches the signature the wiring expects, because both were emitted from the same AST node when the protected file was first created.

This is the right shape for the human/generated boundary, and it is worth saying why explicitly. The two unsatisfying alternatives are *no escape hatch* and *unbounded escape hatch*. No escape hatch means every line of every action body must be expressible as a DSL declaration, which would either bloat the DSL with imperative constructs the rest of the architecture rejects or eliminate any action that needs to call a third-party HTTP API. Unbounded escape hatch means letting developers edit any generated file freely, which destroys the coherence contract — the next regeneration either overwrites their edits or fails to maintain consistency with them. The protected-file marker is the narrow middle: human-written code lives in its own files, marked explicitly, and the compiler treats those files as immutable inputs rather than as outputs. Knuth's literate-programming tradition recognized a similar separation between the prose a human writes and the code the system extracts, but Agicore takes the opposite formal stance — separate files rather than woven files — because separate files are easier to grep for, easier to enumerate in a build script, and easier to reason about in version control.

There is a related discipline worth naming, which the compiler implements but the developer rarely thinks about. Generated files carry a marker comment, `// @agicore-generated`, on their first line. This marker has no effect on the build, but it makes the generated/hand-written boundary visible to humans and to tools. A `grep -L '@agicore-generated' src/` listing finds the hand-written files. A code-review tool can hide diffs in generated files by default. A new contributor opening the project can immediately see which files are owned by the compiler and which by the team. The marker is a small piece of metadata that does a large amount of work in establishing the social contract around the codebase.

What gets generated, in full inventory, runs to several megabytes of typed output across a typical application. A SQL migration with all the tables, indexes, foreign keys, and seed data. Per-entity CRUD command implementations in Rust. AI service dispatch code that wires the AI_SERVICE declarations to a multi-provider streaming layer. Per-action stub files (some protected) for IMPL actions. The Tauri main builder that registers every command, opens the database, and configures the window. A Tauri configuration JSON with window dimensions, security CSP, and bundle metadata. A capability ACL that bounds what the application's webview is allowed to invoke. TypeScript interface definitions for every entity, action input, action output, and event. Typed invoke wrappers for every Tauri command. A Zustand store slice for every entity, with state, fetch, create, update, delete actions. A preferences accessor for every PREFERENCE declaration. A list view and a form modal for every entity that has CRUD enabled. An AiChat component for every AI_SERVICE that has a chat surface. The inventory is long, and every line of it is emitted by the same compiler from the same AST, with the same coherence guarantee.

The seventeen specialized generators that the Andon Loop adds in Phase 11 follow exactly the same pattern, applied to a different problem space. They emit mutation-policy parsers, tier verifiers, sandbox harnesses, NBVE evaluators, approval-chain orchestrators, ledger appenders, hash-chain verifiers, and the file-system mirror sink that writes per-ledger `.jsonl` archives for off-database compliance integration. Each generator consumes the AST node corresponding to its declaration — `MUTATION_POLICY`, `ANDON_RESPONDER`, `IMPROVEMENT_REASONER`, `AUTHORITY`, `LEDGER` — and emits typed Rust that participates in the same `cargo build` verification as the rest of the back-end. The Andon Loop generators are not an afterthought layer; they are peers of the CRUD generators, written in the same style, tested with the same fixture discipline, subject to the same cross-generator coherence contract. The architectural addition that the Andon Loop represents is, from the compiler's point of view, seventeen more functions of the AST that emit more typed output files. The discipline scales because the discipline is the same.

It is worth pausing on what is not in the inventory, because the absence is structural. The compiler does not emit business-specific UI design. It does not emit application-specific algorithms beyond CRUD. It does not emit complex domain-specific React components — those are the developer's territory, written by hand, hooked into the generated store. The line between generated and hand-written is not arbitrary; it is the line between wiring and policy. The wiring is mechanical, deterministic, error-prone when written by hand, and identical from one application to the next. The compiler eats it. Policy — what the application should actually do, in terms of business logic and domain-specific user experience — is what the developer is left with. This is the proper division of labor. The compiler removes the work that should not have been work; it does not remove the work that is the actual job.

Robin Milner's slogan that well-typed programs cannot go wrong was, in its original 1978 setting, an argument about the soundness of the ML type system. Applied to code generation, it acquires a slightly different meaning. The generated programs in Agicore are well-typed because two independent compilers checked them after the generators ran. They cannot go wrong in the specific sense that all wiring between layers — database, Rust commands, IPC, TypeScript wrappers, Zustand store, React components — is type-coherent by construction. They can still go wrong in the sense that the application's business logic might be incorrect, the user might enter unexpected input, the third-party API might return surprising data. Those are application-level concerns and are addressed by TEST declarations, by application-level test suites, by the disciplines of Cognition Systems Engineering applied to operating the running system. What the compiler guarantees is that none of those concerns hide behind a wiring error. Every bug you find is a real bug. The compiler has eliminated the false bugs structurally.

A short note on testing the generators themselves is in order, because the test suite is the artifact that lets the generators evolve safely. Each generator function is exercised by a battery of tests whose form is consistent: a small AST fragment is constructed in TypeScript, the generator function is called against it, and the output string is compared against an expected fixture. The fixtures are checked into the repository as ordinary files; when a generator is modified, the changed fixtures appear as part of the pull request and are reviewed alongside the generator change. This is the same discipline the LLVM project applies to its `FileCheck`-based codegen tests and the same discipline GCC applies to its codegen suite. The discipline is not novel; what is novel, for an AI-augmented system, is that the fixtures themselves become reviewable artifacts that document precisely what the compiler currently emits. An AI agent modifying a generator must produce a fixture diff that a human can read. The fixtures are the audit trail of the compiler's behavior.

The discipline extends to backward compatibility. A change to a generator that alters the output for a previously-valid AST is, by definition, a breaking change for every committed project that contains the affected pattern. The compiler authors are careful about such changes — when they happen, they happen explicitly, with a documented migration, and with the understanding that every project depending on the compiler will see the diff on the next regeneration. Most generator changes are additive: a new node kind is supported, a new modifier is honored, a new output file is emitted alongside the existing ones. Additive changes do not break existing projects. The fixtures for the existing patterns remain unchanged. The discipline is the same one any library author with a serious user base applies: changes that affect existing users get the version-number weight that signals their impact. Compiler-output changes are no different in kind, just more pervasive in scope. The fixture suite is the mechanism that makes the impact visible.
