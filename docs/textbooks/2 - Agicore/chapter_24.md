# Chapter 24: Skill Docs — AI Co-authoring at Industrial Scale

The previous three chapters showed Agicore applications running. None of them addressed the question of how those applications get authored when the practitioner authoring them is not the framework's designer and the AI assistant doing most of the typing is not a frontier model with a hundred-thousand-token context budget and a long memory of the project's prior decisions. The question is not academic. The architecture's claim that AI lives at the edit boundary depends on AI authoring `.agi` files competently. Competence depends on the AI assistant knowing the grammar, knowing the idioms, knowing what compiles and what does not, knowing the failure modes that a stricter parser will reject, and knowing how to recover when the parser does reject a draft. A frontier model with the full Agicore documentation in context can do this. A seven-billion-parameter open-source model running on a developer's laptop cannot — unless something else closes the gap. The Skill Doc is that something else. It is the artifact that makes AI co-authoring of Agicore — and, by the conventions the Skill Doc format establishes, AI co-authoring of any disciplined technical domain — viable at industrial scale.

The Skill Doc as Agicore deploys it is a dichotomy. There are two documents, intentionally separate, intentionally addressing different model capabilities and different task scopes. The Baby Step Skill Doc — `agicore.baby.skill.md` — fits in a context budget between roughly seven thousand five hundred and sixteen thousand tokens. It is sized to load alongside a meaningful working context (a partial draft, recent compiler error output, the application being authored) inside the working memory of a model from the seven-to-ten-billion-parameter class. Claude Haiku 4.5, Llama 3.1 8B, Qwen 2.5 7B, Gemma 2 9B, and Mistral 7B are the target models. The scope is small-to-medium desktop applications — the kind of application a practitioner might describe as "build me an app that does X" without specifying the integration topology of a multi-app enterprise suite. The Super Skill Doc — `agicore.super.skill.md` — runs between eighteen thousand and one hundred thousand tokens. It targets Claude Opus 4.7, Claude Sonnet 4.6, GPT-5, and Gemini 2.5 Pro — models with the context budget to hold the full canonical syntax for all fifty-eight declarations, the full anti-pattern catalog, and the full library of worked recipes simultaneously with the project under construction. The scope is full-spec Agicore — multi-app suites, the Andon Loop, mutation policies with multi-signer approval chains, the cross-application packet substrate.

The Baby Step's internal structure is a five-layer organization labeled L0 through L4 — a convention the Skill Doc format codifies and the surrounding ecosystem adopts. The ordering is not cosmetic. Each layer presupposes the one before it, and each layer addresses a distinct failure mode that an AI assistant lacking that layer would otherwise exhibit. L0 is when-to-use: a short prose passage that helps the model decide whether this Skill Doc is the right one for the current task. The model reading L0 should be able to determine, in roughly two hundred tokens, whether the work in front of it is in scope. L0 explicitly enumerates the negative scope as well as the positive scope — the "do NOT use this skill for" cases. The Baby Step's L0 names mobile application development, raw Rust systems programming, and procedural business logic that does not fit a declarative-rules shape as out-of-scope. The negative enumeration is doing real work. Without it, a model handed a vague task ("build me an inventory tracker for iOS") would attempt to apply Agicore's vocabulary to a domain Agicore does not cover, producing fluent but invalid output. The L0 prevents the misapplication structurally; the model is told, before it does anything else, what this skill is not for.

L1 is the mental model — the architectural framing that lets the model understand what Agicore is, what it is for, and what the central commitments are. The distinction the L1 holds is between what the system is and what the system does. The model needs the former to write code; documentation of the latter is L2's job. L1 in the Baby Step covers the inversion (AI at the edit boundary, determinism at runtime, DSL as constraint boundary), the compilation pipeline shape (parser, validator, generators, two-compiler property), and the file format conventions (one `.agi` file per application, `@agicore-generated` and `@agicore-protected` markers). The mental model is approximately fifteen hundred tokens — enough for the model to reason about what it is doing, brief enough not to consume the working context.

L2 is the compressed reference — a tightly packed presentation of all fifty-eight declarations in roughly the form a working programmer needs to write them. The compression is the discipline. Each declaration appears with its keyword namespace, its required and optional fields, its valid value types, and one short canonical example. There is no expository prose between declarations; there is no theoretical justification for design choices; the model reading L2 is treated as a working programmer who needs reference material, not as a student who needs an introduction. L2 in the Baby Step is approximately thirty-five hundred tokens. L3 is the anti-pattern catalog — the failure modes the parser will reject, organized as paired examples of incorrect code and corrected code with a short note on why the correction is required. L3 is the most operationally valuable section because it preemptively rules out the failure modes the model would otherwise discover by submitting drafts to the parser and reading error output. The Baby Step's L3 covers perhaps fifteen common failure modes — wrong keyword ordering inside a declaration, snake_case used where PascalCase is required, fields declared with unresolved types, references to entities not declared elsewhere in the file.

L4 is the worked-examples library — five complete short applications that exercise the most common patterns. Each example is a complete `.agi` file that parses and compiles, sized to be readable end-to-end in two or three minutes. The Baby Step's five examples cover the patterns the model is most likely to be asked to produce: a single-entity CRUD application with one custom action and one generated view; a multi-entity application with foreign-key relationships and a list-detail view pair; an application using an `AI_SERVICE` and a streaming action; an application using a `WORKFLOW` and a `STAGES` block; an application using a `PIPELINE` with `QC` nodes. The worked examples are the model's copy-and-modify substrate. Given a task, the model identifies the worked example closest in shape, copies it, and modifies it to match the task. The parser then arbitrates. The five layers compose, and the composition is itself the proof of the format. L0 narrows the task space; L1 supplies the conceptual frame; L2 supplies the vocabulary; L3 supplies the failure-mode patches; L4 supplies the structural templates. A model with all five in context has everything required to author Agicore. The order is the curriculum.

The mechanical self-check is the load-bearing innovation of the Skill Doc format. The Baby Step is built around the explicit understanding that the parser is the arbiter — that the model does not need to ask itself "is this good code" because the parser will tell it whether the code is structurally correct. The self-check loop is not a rubric the model evaluates against itself with its own judgment. It is a tool invocation. The model writes a draft, invokes the parser, reads the line-and-column error output, identifies the violated rule from L3 or the declaration's syntax from L2, edits the draft, and re-invokes the parser. The loop terminates when the parser accepts the draft. The model is not asked to evaluate whether the code is good. The model is asked to iterate against a deterministic oracle. This is what makes the Baby Step usable by a seven-billion-parameter model. The model does not need to be smart enough to evaluate Agicore quality; it needs to be smart enough to read parser errors and apply patches from the anti-pattern catalog. That capability is well within the seven-billion-parameter class.

The distinction between mechanical and rubric self-check is epistemologically deep and worth developing in its own right, because the Skill Doc format codifies both modes and the choice between them is the central design decision in adapting the format to a new domain. Mechanical self-check is a yes/no decision rendered by a deterministic oracle. The parser either accepts the input or rejects it; there is no in-between, no partial credit, no judgment call. The model invoking the parser is not asking the parser's opinion; it is querying a function whose return value is a Boolean and whose failure modes are enumerable. Rubric self-check is something else entirely. A rubric is a checklist of considerations the model evaluates substantively, using its own judgment as the proxy for what an expert reviewer would say. "Does the mutation policy match the institutional authority structure?" is not a question with a Boolean answer; it is a question whose answer is a paragraph of reasoning that a human reviewer will weigh against context the model does not have. Each mode has its proper domain, and confusing one for the other is the failure mode the Skill Doc format is designed to prevent. Agicore-authoring is mechanical because the DSL is a constraint-satisfaction problem — every property the parser checks reduces to grammar conformance, type compatibility, or reference resolution, and all three are decidable. Accelerando-deployment, by contrast, is rubric because deployment decisions involve judgment that cannot be parsed: whether the approval chain has the right people, whether the regression suite reflects production reality, whether the rule priorities encode the operational urgency the institution actually faces. No parser exists for those questions. None should. They are the human reviewer's questions, and the rubric's job is to surface them in a structured way so the model's draft can be reviewed against them. The Skill Doc format names the mode explicitly in the YAML frontmatter so the consuming model knows which kind of loop it is in. A mode mismatch — a model treating a rubric self-check as if it were mechanical, or treating a mechanical self-check as if it required substantive judgment — produces the kind of failure that frontier-model AI authoring has been embarrassed by repeatedly: confident, fluent, structurally wrong output, or hedged, judgment-laden output where a Boolean answer was sufficient. The mode declaration in the Skill Doc frontmatter prevents the mismatch architecturally.

The Super Skill Doc extends this structure rather than replacing it. The same five-layer shape applies; what changes is fidelity. L0 and L1 in the Super are roughly the same length as in the Baby Step — the model loading the Super does not need more architectural framing than the model loading the Baby Step, only a wider working space for the technical layers underneath. L2 in the Super expands dramatically: all fifty-eight declarations are presented with their full canonical syntax (versus the Baby Step's compressed form), every field documented, every valid value type enumerated, every optional clause explicitly named, and a representative example for each declaration's most common variants rather than a single illustrative example. The expansion is approximately ten thousand additional tokens beyond the Baby Step's L2. The compression that defined the Baby Step — pack each declaration into the smallest faithful form a working programmer can use — is relaxed in the Super; faithfulness now includes completeness, not just functional sufficiency. L3 in the Super expands to twenty-plus anti-patterns (versus the Baby Step's roughly fifteen), covering the failure modes that arise specifically in multi-app suites — the cross-application packet declarations that the framework will reject if the channels do not declare matching delivery semantics, the mutation policies that the tier verifier will reject if the scope kinds are not in any tier, the approval chains that the runtime will refuse to instantiate if the identity references do not resolve, the regression suite specifications that the runtime will reject if the historical traffic samples cannot be replayed against the new rule version. L4 in the Super contains twelve worked recipes (versus the Baby Step's five), each one a complete deployment template for the kinds of applications the Accelerando suite represents — an enterprise ERP module, a clinical decision support application, an Andon Loop deployment with three approval tiers, an integration packet substrate connecting three applications, a regulatory submission application with hash-chain attestation, a procurement workflow with multi-signer approval gates. The scaling pattern is uniform: same shape, different fidelity. A reader who has internalized the Baby Step recognizes the Super on sight, because the Super is the Baby Step with every layer dilated. The Skill Doc format scales by adjusting one parameter — the fidelity-versus-compression tradeoff — while preserving the architectural commitments that make the format usable in the first place.

The Super Skill Doc also includes a section of twelve self-check prompts — explicit invitations to the model to re-verify particular properties of the draft before declaring the work complete. The self-check prompts are not the substitute for the parser; the parser remains the arbiter of structural correctness. The self-check prompts cover the properties the parser cannot prove from declarations alone — that the mutation policy's tier semantics match the institutional authority structure the application is for; that the approval chain's identities are the right people for the kind of mutation being authorized; that the regression suite specification matches the operational reality of the production traffic the application will see; that the rule priorities encode the urgency the operational SLAs require. These are rubric self-checks — structured questions the model answers in prose, with the answers serving as the substrate for the human reviewer who will read the deployment proposal before approving it. The combination is the right division of labor: mechanical self-check for structural correctness, rubric self-check for substantive correctness, human review for institutional appropriateness.

A worked example clarifies how the Skill Docs operate together. Suppose an AI assistant — running a seven-billion-parameter open-source model on a developer's laptop — is asked to author a HOC-style home-monitoring application. The assistant loads the Baby Step into context and reads L0 to confirm scope. The task is in scope. The assistant reads L1 to confirm the mental model is appropriate — yes, this is a single-application Agicore project. The assistant reads L4 to find the closest worked example. The closest is the workflow-with-stages example, but the task requires modules with `EXPECTS_MATCH true`, which is a pattern the Baby Step's L2 mentions but does not include a full worked example for. The assistant drafts what it believes is correct, invokes the parser, and reads the error output. The parser rejects the draft at line 47, column 12, with a message identifying that the `MUTATION_POLICY` block's `TIER 2` declaration has a `SCOPE` value the verifier cannot resolve to any tier's coverage. The assistant consults the Baby Step's L3 anti-patterns for `MUTATION_POLICY` errors. The anti-patterns address the most common cases but do not address the specific case the assistant has produced. The assistant escalates — it requests the Super Skill Doc be loaded into context. The Super Skill Doc's L3 has the specific failure mode the assistant hit: `SCOPE` declared with a scope kind that no tier in the policy covers, which the verifier will treat as the not-in-any-tier pattern and reject. The Super's L4 has a worked recipe for a three-tier mutation policy with the scope kinds correctly distributed across tiers. The assistant copies the recipe's structure, adapts it to the application, re-invokes the parser, and the draft parses. The compiler then runs, the application builds, the assistant declares the work complete, and the human reviewer reads the result before deploying. The entire authoring loop closed without the assistant's own judgment being the load-bearing source of correctness. The parser was the oracle. The Skill Doc was the reference. The human was the institutional reviewer.

What the Baby Step demonstrates as a capability statement is that a seven-billion-parameter open-source model — running locally, with no API call, with no per-request cost, with no dependency on a frontier-model vendor's availability — can author valid Agicore. The capability is asymmetric in an architecturally important direction. The model is not authoring fluent prose, analyzing legal contracts, or generating creative work — tasks where seven-billion-parameter models are visibly worse than frontier models. The model is performing constraint-satisfied code generation in a DSL whose grammar is finite, whose anti-patterns are enumerated, and whose verification is mechanical. The constraint satisfaction reduces the search space to what the model can handle. A seven-billion-parameter model asked to write idiomatic Python over an arbitrary problem domain searches across every library and architectural choice a fluent Python program might make; the model's relative weakness shows up as plausible-but-wrong choices among the many that compile. The same model asked to write Agicore against the Baby Step searches only the space the Baby Step admits: fifty-eight declarations, a handful of recipes, an enumerated set of anti-patterns to avoid. The search space is small enough that the model's competence within it is high. The Skill Doc, in effect, multiplies the model's effective capability by an order of magnitude in this domain — not by making the model smarter, but by collapsing the space the model is searching to one it can navigate competently. Most of what we call "ability" in a disciplined technical domain is, on close inspection, knowing what is admissible. The Skill Doc encodes admissibility. The model supplies the local edits.

The implication is that the population of practitioners who can deploy Agicore applications expands dramatically. A developer who can run a seven-billion-parameter model on their laptop has the AI authoring substrate they need. The cloud frontier is not required for the authoring loop, only for tasks where the cloud frontier's competence is genuinely necessary — Super Skill Doc territory, multi-app suites, deep integration topologies. The Baby Step does not eliminate frontier models from the workflow; it eliminates them from the workflow's critical path. Solo developers, regional health systems, mid-sized manufacturers, municipal IT departments — institutions for which frontier-API costs and vendor dependencies are real constraints — can deploy Agicore competently because the Baby Step makes their available AI substrate sufficient for the work.

The Skill Doc format is not Agicore-specific, and treating it as a portable convention is the second contribution this chapter makes. The format — L0 when-to-use, L1 mental model, L2 compressed reference, L3 anti-patterns, L4 worked examples, plus an explicit self-check mode (mechanical or rubric) — applies to any technical domain where a model needs to behave as a competent practitioner against a stable specification. Tauri ACL configuration could ship a Baby Step and a Super Skill Doc. Rust ownership and borrowing could ship a Baby Step and a Super Skill Doc. Kubernetes resource manifests could ship a Baby Step. SQL injection prevention could ship a Baby Step with parser-as-oracle defined as a static analysis tool. ERP deployment consulting — the kind of work currently performed by humans charging consulting day-rates against SAP or Oracle deployment templates — could ship a rubric-self-check Baby Step where the rubric is the deployment checklist that experienced consultants currently carry in their heads. The format is a portable convention for packaging "teach an AI assistant to do this domain well" as a single Markdown file with YAML frontmatter, with the self-check mode explicitly named so the consuming AI knows what kind of verification loop it is in.

The portable convention's deepest claim is that the format addresses the structural problem of AI authoring at scale. The structural problem is not that AI is bad at any given task. The structural problem is that any given AI assistant — even a frontier model — does not know, by default, what the institutional discipline of a domain is. The AI knows English. The AI does not know the difference between an HL7 v2 ADT^A04 message and an ADT^A08 message, why the radiology critical-finding window is sixty minutes rather than thirty or one hundred and twenty, why the controlled-substance PDMP query must precede the dispense rather than follow it, or what the appropriate approval chain is for a clinical decision support rule modification at a community hospital versus a national academic medical center. These are institutional knowledge questions whose answers live in the human practitioners and the regulatory documents and the accumulated experience of the domain. The Skill Doc is the artifact that captures the institutional knowledge in a form the AI can consume. The Baby Step captures the minimum viable institutional knowledge for the task profile. The Super Skill Doc captures the comprehensive institutional knowledge for the deeper task profile. Together they constitute an industry's transferable practical wisdom, packaged for AI co-authoring.

The CSE-textbook angle on the Skill Doc is direct. Cognition Systems Engineering, as the companion volume establishes, is the discipline of designing production systems where AI is one component among several — humans, deterministic software, quality control infrastructure, workflow orchestration, governance machinery. The discipline needs an artifact that lets AI participate competently in the disciplined work. The Skill Doc is that artifact. It is what makes "AI as disciplined co-author" operational rather than aspirational. The mechanical self-check is the property that makes the discipline transferable — the parser is the same arbiter whether the model is a frontier model or a seven-billion-parameter local model, whether the practitioner is a senior architect or a new hire, whether the application is the canary or the eighteen-app enterprise suite. The discipline of the field crystallizes around the artifact. Mechanical engineering's discipline crystallized around CAD drawings. Software engineering's discipline crystallized around the version-control-plus-compile-plus-test loop. Cognition Systems Engineering's discipline crystallizes around the DSL-plus-Skill-Doc-plus-mechanical-self-check loop. Agicore is the first generation of that crystallization. The Skill Doc is the artifact that lets the crystallization include AI as an active participant rather than a passive object of governance.

The principle the Skill Docs validate is the principle that AI co-authoring is solvable when the discipline of the domain is packaged as a constraint specification that the model can consume and the model's output can be mechanically verified against. The packaging is the Skill Doc. The constraint specification is the grammar. The verification is the parser. The combination scales because the model does not need to be smart enough to do the discipline; the model needs to be smart enough to apply the patterns the Skill Doc documents and to iterate against the oracle the parser provides. This is what AI co-authoring at industrial scale looks like. It does not look like one frontier model authoring everything for everyone. It looks like a portable format that lets any AI assistant become a competent practitioner of any disciplined domain, with the discipline encoded as documentation the AI consumes and verification the AI invokes. The reference implementation is Agicore. The convention is portable. The crystallization is underway.
