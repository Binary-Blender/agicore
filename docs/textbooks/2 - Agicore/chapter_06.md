# Chapter 6: Grammar, Lexer, Parser

A grammar is a theory of which sentences are legal. In a programming language, that theory is also a theory of which programs the discipline considers worth writing. Wirth, who designed Pascal, Modula, and Oberon while writing the textbook that taught a generation of compiler engineers, argued that language design is fundamentally data-structure design: the syntax describes the shape of the tree the parser will construct, and the tree's shape determines what the rest of the compiler can do cheaply. Designing a language well, on this view, is the same activity as designing a database schema well — pick the wrong representation and every operation downstream becomes a workaround. Pick the right one and the operations become obvious.

The Agicore grammar is small, by the standards of the language families it borrows from. There are fifty-eight top-level declarations and a handful of expression forms. Every declaration is a keyword followed by an identifier and a braced block; every block is a sequence of fields, each of which is a keyword followed by a value. There are no precedence tables to argue about, because there are no operator expressions to speak of. There are no statement forms to design, because the language has no procedural body — it declares structures and the compiler emits the procedures. Backus's BNF, which gave the field its first formal vocabulary for describing grammars in 1959 in the ALGOL 60 report, would render the entire Agicore declaration grammar in perhaps a hundred and twenty production rules. Most of the engineering attention does not go into the grammar's surface. It goes into deciding what each declaration's body is allowed to contain — and that decision is the central question Chapters 9 through 15 will treat layer by layer. This chapter is about the surface: the lexer, the parser, the AST, and the conventions that make the surface stable enough to support the rest.

Begin with the lexer. A lexer's job is to turn a string of characters into a stream of tokens — atomic syntactic units that the parser can dispatch on without rescanning text. The Agicore lexer lives in a single TypeScript file, `core/parser/src/lexer.ts`, and is a few hundred lines long. It is hand-written. There is no `lex` or `flex` involvement, no parser generator, no DSL-of-DSLs. The choice deserves brief defense. Aho, Sethi, and Ullman's dragon book makes the canonical case for generated lexers: regular expressions are easier to read than tokenization code, generators produce optimized state machines, and the regularity of the specification makes correctness easier to argue. None of these arguments is wrong. They are simply not strong enough to justify the indirection in this particular system. The keyword set is the discipline of the language. Adding a keyword is the same act as adding a primitive to the field. The keyword map should be read and edited by humans directly, in the same file where the act of tokenization occurs, so that what is admissible in the syntax is legible in one place. The cost of hand-maintenance for a few hundred keywords is small. The cost of having the language's vocabulary live behind a generator is non-trivial in cognitive overhead. The compiler chooses the cheaper of the two costs.

The lexer's data structure is a `Map<string, TokenKind>`. When the scanner encounters a sequence of identifier characters, it builds the identifier string, looks it up in the map, and if it finds a match emits the corresponding keyword token; if it does not, it emits an `IDENT` token carrying the string. This is the trick that lets soft keywords work — a word like `state` can be a keyword in one position and an identifier in another, because the parser decides at the call site whether to demand the keyword form. Phase 10 of the project, building `novasyn_mba.agi`, surfaced a useful lesson about this design: before adding a token, grep the keyword map. Most of the keywords that felt missing turned out to be present already, tokenized correctly, but not yet wired into the `parseFoo` function for the declaration that needed them. The work was at the parser layer, not the lexer layer. The lesson generalizes: in a hand-maintained lexer with a keyword map, the lexer is rarely the bottleneck. The parse functions are.

Whitespace is consumed and discarded. Comments are recognized. Two comment forms matter: the ordinary `//` line comment, which is discarded, and the `// @agicore-protected` marker, which the parser records so that generators downstream can decide whether to overwrite a file. String literals are bounded by double quotes and support a minimal escape set. Numeric literals are scanned as decimal integers and floating-point values. Punctuation tokens — brace, bracket, parenthesis, comma, colon, semicolon, equals, arrow `->`, double-pipe `||`, single pipe `|` — are each their own token kind. Line and column counters are maintained as the scanner moves left to right, and every token carries its source position so that error messages later in the pipeline can point at a specific character.

The token stream now feeds the parser. The parser is recursive-descent, the simplest workable approach for a grammar of this shape and the approach the compiler-engineering tradition has favored for hand-written front ends since the 1960s. The top-level structure is a loop: read the next non-whitespace token, dispatch on its kind, call the corresponding `parseFoo` function, append the returned AST node to the program. Each `parseFoo` is a short function whose structure mirrors the production rule for that declaration. `parseEntity` consumes the `ENTITY_KW`, expects an identifier and binds it, expects a left brace, loops over field declarations until it sees the right brace, returns an `EntityDecl` node. `parseAction` does the same for actions, with a more elaborate body parser to handle the four action forms — `AI`, `IMPL`, `PATTERN`, `EMIT` — and the input/output contract. `parseWorkflow` parses steps and dependency lists. `parseRule` parses both classic and modern rule forms, accepting `IF` as an alias for `WHEN` and synthesizing actions from `FLAG` declarations.

A short grammar fragment, in a BNF that hews close to the actual parser dispatch, captures the entity-and-field portion:

```
program       ::= decl*
decl          ::= entity_decl | action_decl | workflow_decl | rule_decl | ...
entity_decl   ::= 'ENTITY' IDENT '{' field_or_mod* '}'
field_or_mod  ::= field_decl | modifier_decl
field_decl    ::= IDENT ':' type_ref field_attr*
type_ref      ::= IDENT ('|' IDENT)*
field_attr    ::= 'REQUIRED' | 'UNIQUE' | 'DEFAULT' literal
modifier_decl ::= 'TIMESTAMPS'
                | 'CRUD' crud_ops
                | 'BELONGS_TO' IDENT
                | 'HAS_MANY' IDENT
                | 'SINGLETON'
                | 'STAGES' '[' IDENT (',' IDENT)* ']'
```

This fragment is small but already encodes several decisions worth examining. The `type_ref` admits a pipe to express union types. The compiler reads `string | null` as the optional-string type and maps it to Rust's `Option<String>` and TypeScript's `string | null`. The presence of the pipe in the grammar dates to Phase 8 of the project, when building `benders_killer_app.agi` exposed that an action's output type could be legitimately absent — a search that returns nothing, a parse that may fail. The PIPE token was added to the lexer; the parser learned to read the pipe form; the Rust generator learned to emit `Option<T>`; the TypeScript generator learned to emit the union. One primitive moved through every layer in one session. The pattern is the framework-evolution discipline of Chapter 25 in miniature.

The choice of braced blocks rather than significant indentation is also worth defending briefly. Python and YAML show that indentation-significant syntax can be readable and writable; they also show that indentation is fragile under copy-paste, hostile to mechanical rewriting, and ambiguous when whitespace is invisible. Agicore must be writable by both humans and AI agents, and pasted between contexts with different tab widths. Braces are robust. The cost is two characters per block, which is negligible in a declarative language where blocks are short.

When the parser finishes, the result is an AST: a tree of typed TypeScript objects whose shapes are declared in `core/parser/src/types.ts`. The AST is the contract between the front end and everything downstream. Every code generator consumes the AST. The static validator consumes the AST. The pretty-printer consumes the AST. Tools that lint, format, document, or analyze `.agi` files consume the AST. As long as the AST's shape is stable, the surface syntax can change without breaking downstream consumers. This is the same separation of concerns that LLVM's intermediate representation gave to the C and C++ compilers — the front ends can be replaced, the back ends can be replaced, but the IR holds the contract. In Agicore the AST plays the same role.

The AST for our running example is approximately:

```typescript
{
  kind: 'program',
  declarations: [
    {
      kind: 'entity_decl',
      name: 'Student',
      sourcePos: { line: 1, column: 1 },
      fields: [
        {
          kind: 'field',
          name: 'name',
          type: { kind: 'type_ref', name: 'string', nullable: false },
          attrs: [],
          sourcePos: { line: 1, column: 19 }
        }
      ],
      modifiers: [],
      stages: null
    }
  ]
}
```

The source positions are not decoration. They are what allows the static validator to emit errors pointing at the developer's exact column, and what allows the pretty-printer to preserve user comments and formatting through round-trips when desired. The `kind` discriminator on every node is what allows TypeScript's discriminated-union typing to verify exhaustively in the generators — when a generator switches on `decl.kind`, the type system refuses to compile the generator until every kind is handled or explicitly punted to a default. The compiler uses the language it is written in to enforce that the compiler itself handles every shape of program it can accept.

A short illustration is useful at this point. Take the action declaration form, which is one of the more elaborate productions in the grammar because it must accept four substantially different action forms — `AI`, `IMPL`, `PATTERN`, `EMIT` — under a unified declaration syntax. The relevant grammar fragment reads, in BNF approximation:

```
action_decl   ::= 'ACTION' IDENT '{' action_body '}'
action_body   ::= action_inputs action_outputs action_kind
action_inputs ::= ('INPUT' IDENT ':' type_ref ('REQUIRED')?)*
action_outputs::= ('OUTPUT' IDENT ':' type_ref)*
action_kind   ::= ai_kind | impl_kind | pattern_kind | emit_kind
ai_kind       ::= 'AI' STRING
impl_kind     ::= 'IMPL' STRING
pattern_kind  ::= 'PATTERN' ('file_handler' | 'shell_open')
emit_kind     ::= 'EMIT' IDENT '{' (IDENT ':' type_ref)* '}'
```

`parseAction` reads the leading keyword and identifier, then enters the body parser, which collects input declarations, output declarations, and dispatches to one of four `parseFooKind` functions based on which kind keyword it sees next. Each kind produces a different shape of AST node, but the surrounding wrapper — name, inputs, outputs — is shared. The Rust generator dispatches on the kind to emit different implementations; the TypeScript generator does the same on its side. The four forms coexist under one keyword because the field has decided that the four forms are variants of one concept — *do something the framework will package as a typed callable* — rather than four separate concepts requiring four separate declarations. The grammar encodes the field's decision about what is conceptually unified.

Naming conventions are part of the grammar, in the precise sense that the parser refuses to accept names that violate them. ENTITY, PACKET, MODULE, SKILL, and other type-style declarations require PascalCase names — `Student`, `InvoiceLine`, `OrderHeader`. ACTION, WORKFLOW, RULE, and other verb-style declarations require snake_case names — `create_student`, `onboard_user`, `flag_stale_article`. Fields are snake_case. Keywords are ALL_CAPS. SQL columns are snake_case. TypeScript fields are camelCase. The compiler enforces the snake_case-to-camelCase conversion at the Rust-TypeScript boundary, deterministically, without involving the developer. The conventions are not aesthetic. They are the convention by which the same conceptual identifier appears in five different forms across the generated stack — Rust struct field, SQL column, JSON serialization key, TypeScript interface field, React form input name — without anyone having to think about which form is required where. The grammar refuses to accept malformed names because malformed names break the invariant.

This is, in essence, the same insight Conway named in 1968 about software organizations: the shape of the system that gets built mirrors the shape of the communication structure that builds it. Apply the law to AI-assisted authoring. The DSL is the communication structure between the human author, the AI assistant, the static validator, and the generators. Naming conventions encoded in the grammar are how the structure stays coherent across the many participants in the authoring loop. A name that conforms to the convention can be transformed mechanically through every layer; a name that violates the convention requires special handling somewhere, which is the place future bugs will appear. The grammar refuses the special case.

Soft keywords deserve a brief note because they are the trick that lets a fifty-eight-declaration grammar coexist with a finite human vocabulary. A soft keyword is an identifier that the lexer always tokenizes as `IDENT`, but that the parser, when it is parsing the body of a specific declaration, will accept as if it were a keyword. The state names in a `STAGES` declaration are the canonical example: `STAGES OrderLifecycle { draft -> submitted -> approved }` uses `draft` and `submitted` and `approved` as state-name identifiers, even though those words might be keywords or fields elsewhere. The technique is safe because the parser knows what position it is in; it is necessary because the alternative — reserving every state name a developer might want as a keyword — would consume the identifier namespace in absurd ways. Soft keywords are how the grammar admits a long tail of domain-specific names without forcing the lexer to know about them.

Error messages from the parser have one job, which is to point at the source position of the offending token and explain what was expected. Recursive-descent parsers do this naturally because every `parseFoo` knows what production it is in. When `parseEntity` is reading field declarations and encounters a token it does not recognize, it can emit a message of the form *expected field name, modifier, or `}`, found `IF` at line 12, column 5*. The discipline is precise diagnostics, not heroic recovery; if the parser cannot continue, it stops, and the developer fixes the problem before re-running. This is the right discipline for a language whose authoring loop is AI-augmented, because the AI agent can iterate against the parser's verdict directly. The parser is the mechanical oracle that an AI authoring loop relies on. A precise error message at the AI's terminal is worth more than a heroic recovery that lets the parser limp through to a second, dependent error.

The parser's error-handling discipline interacts with one practical question worth addressing: what happens when the parser encounters source it cannot recover from in the middle of a large file. The choice is a small one with disproportionate consequences. The Agicore parser does not attempt panic-mode recovery — it does not skip ahead to the next plausible synchronization point and resume parsing in the hope of reporting more errors per run. It stops at the first unrecoverable error and reports that error precisely. The cost is that a file with two unrelated mistakes requires two compile-fix cycles to clean. The benefit is that every error message corresponds to a real error rather than to a downstream confusion caused by misinterpretation of a prior error. In a language whose authoring loop is *write, compile, fix* with sub-second iteration time, the cost is negligible and the benefit — confidence that every diagnostic is genuine — is substantial. The trade is correct for the surface conditions Agicore is built for; a batch-compiled language with hour-long build times would make the opposite trade.

The parser is mature. The current snapshot shows more than five hundred parser tests passing — each one a recorded example of a `.agi` source that should produce a particular AST, or a `.agi` source that should produce a particular error. The test suite is the regression net that lets the parser grow without breaking. Every new syntactic primitive arrives with tests; every parser bug becomes a test before it becomes a fix. Robin Milner's "well-typed programs cannot go wrong" applies here at a level lower than the type system Milner had in mind: well-tested parsers cannot regress, in the sense that a previously-correct parse remains correct forever. The five hundred-test floor is what allows the parser to be modified confidently — what would otherwise be a risky change becomes a routine one, because the cost of breaking something is bounded by the time it takes the test suite to run.

There is a subtler point worth surfacing about the relationship between grammar and AST. A grammar change that preserves the AST is invisible to everything downstream — generators do not need to be touched, validators do not need to be touched, tests of generated output do not need to be updated. A grammar change that alters the AST cascades through the whole compiler. The compiler's authors learn quickly to design changes so that the AST changes as little as possible. Adding the union-type pipe in Phase 8 added one field to the `TypeRef` node — `nullable: boolean` — and required exactly the generators that emit type-specific code (Rust's `Option<T>`, TypeScript's `| null`) to consult that field. Most generators were untouched. The AST as a stable interface is what keeps the cost of language evolution proportional to the depth of the change rather than to the breadth of the compiler.

It is worth comparing the recursive-descent choice against the dominant alternatives, because the comparison illuminates what kind of compiler Agicore is. Parser-generator approaches — LALR-style tools in the yacc/bison lineage, GLR for ambiguous grammars, PEG for greedy combinators, LL(k) for top-down generators — all offer real advantages on languages whose grammars are larger or more ambiguous than Agicore's. ANTLR-class tools handle Java's syntactic surface, with its expression precedence and statement variants and template specializations, more tractably than a hand-written parser could. Tree-sitter handles the incremental reparse for editor integration that a batch-mode hand-written parser cannot match. None of these advantages are inert; they are real and have been chosen correctly by the languages that chose them. The Agicore grammar is simply small enough and declarative enough that the indirection cost of a generated parser outweighs the maintenance cost of a hand-written one. The fifty-eight declarations have similar shapes, the keyword set is finite and slow-growing, and editor integration is a non-goal for now — the authoring loop is text editor plus compiler verdict, not IDE plus live highlighting. The choice is right for the language Agicore is. The choice would be wrong for a language with significantly more syntactic surface.

A related architectural decision is worth surfacing because it explains why the lexer and parser are versioned together with the rest of the compiler rather than packaged separately. The parser depends on the AST type definitions in `core/parser/src/types.ts`. The static validator depends on the same types. The generators depend on the same types. The shared type definitions are the contract that lets the lexer be modified without breaking the validator and the validator be modified without breaking the generators, but only when the AST itself is unchanged. When the AST changes — when a new field is added to `EntityDecl`, when a new node kind is added to the program — every consumer that switches on the affected node kind must be updated, or the TypeScript compiler will refuse to build the compiler itself. This is the compiler eating its own dog food in the most literal sense: the same discriminated-union exhaustiveness that the generated applications rely on is the property that keeps the compiler's internal stages in sync. Modifying the AST is a constrained operation precisely because the language the compiler is written in catches the consequences immediately.

The grammar, the lexer, and the parser are thus three faces of one decision: what the discipline considers worth saying. The grammar is the formal specification. The lexer is the recognizer for the grammar's atoms. The parser is the recognizer for the grammar's productions, and the constructor of the AST that the rest of the compiler consumes. Together they are a few thousand lines of TypeScript that take less than fifty milliseconds to run on a real `.agi` file. They are not impressive engineering by line count. They are impressive by what they make possible: a typed, validated, source-positioned tree that every generator can rely on, and a precise yes-or-no answer for whether a given AI-authored draft is admissible.
