# Chapter 17: MUTATION_POLICY and the Tier Verifier

The previous chapter argued that the Andon Loop's load-bearing property — that AI cannot expand its own authorization — is mechanical rather than procedural. This chapter develops the machinery. A single declaration governs the entire Andon Loop for a given target system: `MUTATION_POLICY`. The declaration names which workflows or modules it governs, enumerates the tiers of mutation it permits, binds each tier to a scope vocabulary and a verification regime, designates the andon responder and improvement reasoner that may originate proposals, and routes audit entries to a named ledger. From that one declaration the compiler emits the proposal table, the tier verifier, the sandbox executor, the regression-suite resolver, the auto-deploy and escalate branches, the approval-request workflow with signature aggregation logic, the SPC gate, the cadence poller, the React mutation console, and the ledger hash-chain helpers — approximately nine thousand seven hundred lines of Rust and TypeScript across seventeen generator files, all consistent with the single declarative source. The reader who has worked through Part III will recognize the pattern; the Andon Loop is the same architectural move applied to operational governance.

The declaration's full syntax appears below. Every field is significant; nothing is decorative. The order of fields within a tier block is free, but the semantic constraints between them are not.

```agi
MUTATION_POLICY <name> {
  TARGETS [<workflow_or_module>, ...]            # which systems this policy governs

  TIER <n> <label> {                              # zero or more tiers, n=1..5 conventional
    SCOPE [<MutationScope>, ...]                  # what kinds of mutations are allowed at this tier
                                                  #   RULES_modify | WORKFLOW_modify | WORKFLOW_add |
                                                  #   WORKFLOW_add_step | MODULE_add | ENTITY_add |
                                                  #   MUTATION_POLICY_modify | SCORE_THRESHOLD | ...
    AUTO_DEPLOY <bool>                            # true => deploy on test pass; false => escalate
    REQUIRE <gate>, <gate> | <gate> AND <gate>    # additional gate names (semantic flags)
    REGRESSION_SUITE <suite_name>                 # e.g. 24h_recent_workflows | 7d_recent_workflows
    MONITORING_WINDOW <duration>                  # post-deploy observation window (telemetry)
    NBVE_WINDOW <duration>                        # shadow-mode duration before promotion
    APPROVAL_AUTHORITY <identity>                 # single signer (1-of-1)
    APPROVAL_AUTHORITY [<id>, <id>, ...]          # multi-signer (N-of-N parallel, any order)
    APPROVAL_AUTHORITY ORDERED [<id>, <id>, ...]  # multi-signer (N-of-N sequential)
  }

  ANDON_RESPONDER <reasoner_name>                 # reactive: invoked when andon fires
  IMPROVEMENT_REASONER <reasoner_name>            # proactive: scheduled via REASONER's SCHEDULE
  LEDGER <ledger_name>                            # routes audit entries to a named log
}
```

`TARGETS` names the workflows or modules whose mutations this policy governs. A single Agicore application can host multiple `MUTATION_POLICY` declarations, each scoping a different region of the system — a permissive policy for the experimental classifier, a restrictive policy for the regulated billing workflow, a maximally restrictive policy for the policy itself. The targets list is the boundary of the policy's authority; a mutation proposed against a target outside the list cannot be governed by this policy and falls back to whichever policy claims the target, or to a global default if none does.

`TIER` blocks are the heart of the declaration. A policy may declare zero or more tiers, conventionally numbered one through five, each with a human-readable label. The label is documentation; the tier number is load-bearing. Inside each tier block, `SCOPE` enumerates the mutation kinds the tier permits. The vocabulary is fixed by the compiler — `RULES_modify`, `WORKFLOW_modify`, `WORKFLOW_add`, `WORKFLOW_add_step`, `MODULE_add`, `ENTITY_add`, `MUTATION_POLICY_modify`, `SCORE_THRESHOLD`, and a few others — and is extensible only through framework evolution. Each scope kind names a category of change the AI can propose. Threshold tuning is a different scope kind than rule addition; rule modification is a different scope kind than rule addition; modifying the mutation policy itself is a distinct scope kind that exists primarily so it can be restricted to the most consequential tier. `AUTO_DEPLOY` is a boolean: when true, a proposal that passes verification deploys without human approval; when false, the proposal escalates to an approval queue. `REGRESSION_SUITE` names the suite of test cases the sandbox replays after applying the proposed mutation; the most common values are `24h_recent_workflows` and `7d_recent_workflows`, which resolve to suites the runtime assembles automatically from production execution logs. `MONITORING_WINDOW` and `NBVE_WINDOW` are durations that govern post-deploy telemetry observation and pre-promotion shadow execution respectively; both will be developed in the next chapter. `APPROVAL_AUTHORITY` declares who must sign off on a proposal that does not auto-deploy, in one of three forms — single signer, parallel multi-signer, or ordered multi-signer — each with distinct semantic guarantees.

The conventional tier semantics are a discipline the compiler does not enforce but the field has converged on. Tier numbers map to mutation classes the way pay grades map to authorization in a hierarchical organization.

```
Tier | Mutation type                       | Auto-deploy?              | Verification
-----+-------------------------------------+---------------------------+-------------------------------------------------
T1   | Threshold tuning                    | Yes                       | Deterministic test plus regression suite
T2   | Rule addition (no modification)     | Yes                       | Deterministic test plus 24h monitoring window
T3   | Rule modification                   | Yes, with shadow run      | Deterministic test plus 7-day NBVE shadow run
T4   | Structural change                   | Human approval required   | Deterministic test plus governance review plus 14-day NBVE
T5   | Architectural change                | Human approval, signed    | Deterministic test plus full design review
```

T1 covers what manufacturing calls load-following parameter adjustment — nudging a `SCORE THRESHOLD`, tweaking a `SCORE DECAY` rate, modifying a `PATTERN` regex within the same capture structure. These are changes whose blast radius is bounded and whose failure mode is recoverable. T2 covers adding a new `RULE` that does not modify existing rules; the addition cannot break behavior that worked yesterday because no existing rule's conditions or actions have changed. T3 covers modifying an existing `RULE` — changing its `CONDITIONS` or `THEN` clauses — which can break existing behavior and therefore requires shadow execution against live traffic before full promotion. T4 covers structural changes: adding or modifying `STAGES` transitions, adding a new `WORKFLOW STEP`, modifying an `ACTION` input or output type. These touch the shape of the system, not just its parameters, and require human governance review. T5 covers architectural changes: adding a new `WORKFLOW`, a new `MODULE`, a new `ENTITY`, or modifying the `MUTATION_POLICY` itself. T5 mutations are rare and consequential; they require signed approval by named governance authorities.

A representative declaration appears below, taken from the Home Operations Center reference application. The HOC governs a small network of devices in a residence — printers, smart hubs, an air-quality monitor — and uses the Andon Loop to evolve its classification rules as new device behaviors appear on the network.

```agi
TIER 1 threshold_tuning {
  SCOPE              [RULES_modify]
  AUTO_DEPLOY        true
  REGRESSION_SUITE   24h_recent_workflows
}

TIER 2 new_classifier {
  SCOPE                [RULES_modify]
  AUTO_DEPLOY          false
  APPROVAL_AUTHORITY   chris
}

TIER 5 governance {
  SCOPE                [MUTATION_POLICY_modify]
  AUTO_DEPLOY          false
  APPROVAL_AUTHORITY   ORDERED [chris]
}
```

Three tiers, three sets of semantics. T1 permits threshold tuning that auto-deploys after the 24-hour regression suite passes; the AI can propose minor adjustments to the network classifier and they reach production without human review, with the regression suite as the mechanical guarantee that nothing that worked yesterday is broken today. T2 permits the same scope kind — `RULES_modify` — but with auto-deploy off and a single named approver, which is the right shape when the AI is proposing a new classifier whose behavior the operator wants to review even though the mechanical tests have passed. T5 permits modification of the mutation policy itself, with no auto-deploy and an `ORDERED` approval chain — even though the chain contains only a single signer in this example, the `ORDERED` form makes the position of the signature explicit and prevents the failure mode in which a later addition to the chain inherits the first signature retroactively. A larger organization would populate T4 with structural-change scopes and T5 with multi-signer ordered chains; the HOC's minimal three-tier shape is appropriate for a single-operator residential system but illustrates every form the syntax supports.

The mechanism that makes the tier system load-bearing is the tier verifier. The verifier is a static gate that runs on every proposal before any test is executed. Its job is precise: enforce that a proposal cannot claim a tier whose scope vocabulary does not cover every scope kind the proposal touches. The algorithm has three steps.

First, for each kind in the proposal's `claimed_scope` set, the verifier searches the policy's tier definitions to find the minimum tier number whose `SCOPE` list contains that kind. The minimum is the lowest-numbered tier that admits the kind. A proposal that touches `RULES_modify` finds, in the HOC example, that T1 admits `RULES_modify` — so the minimum tier for `RULES_modify` is one. A proposal that touches `MUTATION_POLICY_modify` finds, in the same example, that only T5 admits it — so the minimum tier for `MUTATION_POLICY_modify` is five. Second, the verifier takes the maximum of those minimum-tier values across all scope kinds in the claimed scope. The maximum is the `resolved_tier` — the lowest tier number that admits the full set of scope kinds the proposal touches. A proposal touching only `RULES_modify` has resolved tier one; a proposal touching `RULES_modify` and `MUTATION_POLICY_modify` together has resolved tier five, because every kind must be admitted somewhere and the highest required floor governs. Third, the verifier compares `resolved_tier` to the proposal's `claimed_tier`. If `resolved_tier > claimed_tier`, the verifier rejects the proposal with status `tier_rejected` — a terminal state — and writes the rejection to the ledger. The proposal does not reach the sandbox. The proposal does not consume test resources. The proposal does not enter the approval queue. It is dead on arrival.

The verifier's placement in the lifecycle is what makes it impossible to bypass. It runs before the sandbox. The order is cryptographic rather than aesthetic — a verifier that runs after the sandbox could in principle be bypassed by a sandbox special case, by a proposal whose claimed scope is honest but whose runtime effect is broader than declared, or by a clever construction in which the sandbox itself was the verifier. None of these matter when the verifier is the first gate. The sandbox cannot run code that the verifier has already rejected, because the rejection is terminal and the workflow that would have invoked the sandbox does not execute. The verifier cannot be bypassed by a REASONER proposing a mutation with creative framing — the framing does not enter the verifier's calculation, only the claimed scope does, and the claimed scope is structurally bounded by the proposal schema. The verifier cannot be bypassed by a malicious policy modification — modifying the `MUTATION_POLICY` itself requires a T5 mutation, which requires its own approval chain, which is governed by the policy's own tier definitions. The self-modification trap closes itself.

The self-modification trap is worth pausing on. The obvious failure mode of any tier-based authorization system is the meta-proposal: AI proposes "all mutations are now T1," then proposes a destructive change at T1. Without architectural protection, the meta-proposal flows through the same lifecycle as every other proposal and the system's authorization model becomes whatever the most recent meta-proposal said it was. The protection in Agicore is direct: `MUTATION_POLICY_modify` is itself a scope kind in the vocabulary, and the conventional placement is in a single tier — T5 — with the most restrictive approval chain the organization has declared. The verifier treats `MUTATION_POLICY_modify` like any other scope kind. A proposal that touches it has resolved tier five regardless of what the AI claimed. The AI cannot propose its way to expanded authorization, because any proposal that would expand authorization is itself routed to the most restrictive tier the policy declares. The architecture is closed under self-reference. The protection is structural.

The second pattern the syntax supports is even stronger than the self-modification protection. A scope kind that appears in zero tiers cannot be proposed at all. The verifier's first step — find the minimum tier number whose `SCOPE` list contains the kind — returns no result for a kind that no tier admits. The proposal cannot resolve a tier. The verifier rejects with `tier_rejected` and notes that the scope kind is not in any tier of the governing policy. This is the not-in-any-tier pattern, and it is the strongest form of architectural prohibition the system supports. A medical-claims system that wishes to make it mechanically impossible for the AI to propose rules that could systematically deny coverage to a protected class declares the scope kind `DEMOGRAPHIC_FILTER_modify` and leaves it out of every tier in the governing policy. Any proposed rule that could modify a demographic filter is mechanically blocked at the tier verifier — `scope [DEMOGRAPHIC_FILTER_modify] not in any non-empty tier` — and it cannot reach the sandbox regardless of what the model proposes. The mechanism is the same one the self-modification protection uses; the difference is only that no tier admits the kind at all, rather than that only the most restrictive tier admits it. The not-in-any-tier pattern is how an organization makes a thing structurally impossible rather than procedurally prohibited.

The contrast is worth stating sharply. In a continual-harness implementation, "the AI cannot propose discriminatory rules" is a policy enforced by a governance layer. The layer scans proposals, applies a heuristic, and rejects matches. The heuristic has false negatives — proposals whose effect is discriminatory but whose form does not match the pattern the heuristic recognizes — and the heuristic has false positives — proposals whose form matches the pattern but whose effect is benign. The team responsible for the layer maintains the heuristic, tunes it against observed proposals, and is held accountable for failures the heuristic does not catch. In Agicore, "the AI cannot propose discriminatory rules" is a structural property of the policy declaration. The scope kind does not appear in any tier. The verifier does not need a heuristic; it needs only to look up the scope kind in the tier table and observe that no tier admits it. There is no false-negative case because the verifier does not interpret the proposal's content — it inspects the proposal's declared scope, which is bounded by the proposal schema, which is generated from the same compiler that generated the verifier. The architecture closes the loop.

`APPROVAL_AUTHORITY` is the field that governs the human side of the lifecycle for mutations that do not auto-deploy. The three forms are syntactically similar but semantically distinct.

The first form is the single signer: `APPROVAL_AUTHORITY chris`. One named identity signs off and the mutation is approved. This is the right shape when one person owns the system and accountability for changes resides with that person — a residential operator, a single-tenant analyst, a small-team lead. The compiler generates an approval request whose threshold is one of one; the runtime collects a signature, validates the signer against the named identity, and transitions the proposal to approved.

The second form is the parallel multi-signer: `APPROVAL_AUTHORITY [ops_lead, security_lead]`. All named identities must sign in any order. This is the right shape when accountability is shared across roles and the signers are peers — operations and security must both sign on changes to a production system, but neither has authority that depends on the other's prior signature. The compiler generates an approval request whose threshold is N of N, ordered false; the runtime collects signatures from each signer independently and transitions the proposal to approved when the last required signer has submitted. Either signer can reject; rejection is terminal.

The third form is the ordered multi-signer: `APPROVAL_AUTHORITY ORDERED [ceo, cto, board_chair]`. The signers must sign in declared order. This is the right shape when later signers depend on earlier signers for context — the board chair will not sign a proposal the CTO has not yet seen, the CTO will not sign a proposal the CEO has not yet endorsed. The compiler generates an approval request whose threshold is N of N, ordered true, and the runtime enforces the position constraint: signer N cannot sign before signer N minus one. Each signer sees the previous signer's signature and notes, and can veto based on the full context. The pattern's purpose is to prevent the failure mode in which the first signer rubber-stamps every proposal and the remaining signers rely on the first signer's judgment. In an ordered chain, each subsequent signer takes responsibility for what the prior signers have endorsed, which makes the prior signature load-bearing rather than ceremonial.

The signature record is persisted in an `approval_chain` JSON column on the proposal row and includes signer identity, timestamp, signature value, and notes or reasoning. The full chain becomes part of the audit record for the proposal and is hashed into the ledger entry that records the transition to approved. Chapter 19 develops the cryptographic mechanics of the audit chain; the point for this chapter is that the approval record is mechanical evidence rather than procedural attestation. A regulator asking "who approved the mutation that changed the threshold on the credit-screening module" receives a deterministic answer in seconds, with the signature chain and the signers' notes intact.

The remaining fields of `MUTATION_POLICY` route the proposal lifecycle to its operational substrates. `ANDON_RESPONDER` names the REASONER invoked when an andon trigger fires; the REASONER's output is a proposed mutation that enters the lifecycle at the verifier. `IMPROVEMENT_REASONER` names the REASONER scheduled (via its own `SCHEDULE` field) to scan execution logs and propose efficiency mutations proactively; its output enters the same lifecycle. `LEDGER` names the ledger to which audit entries route, allowing an organization to maintain separate ledgers for distinct policies — a permissive ledger for the experimental classifier, a regulatory-grade ledger for the billing workflow, an air-gapped ledger for the policy modifications themselves. Each ledger is independently hash-chained and independently verifiable; tampering with one does not affect the others.

The reader who has followed this chapter through the tier verifier algorithm, the self-modification protection, the not-in-any-tier pattern, and the three approval-authority forms should be able to look at any `MUTATION_POLICY` declaration and predict its operational behavior precisely: which proposals can be auto-deployed, which must escalate, which signers are required for which categories of change, which scope kinds are mechanically blocked, and where the audit trail lands. The declaration is the source code of governance. The compiler generates the verifier, the approval workflow, the signature aggregator, and the ledger hooks. The runtime executes the deterministic logic. The AI proposes; the verifier judges; the sandbox tests; the humans sign when human signature is required; the ledger records every transition. Nothing in the lifecycle requires a policy that someone has to maintain. Everything in the lifecycle is built.

The next chapter develops the verification triad that runs after the tier verifier has accepted a proposal: the sandbox executor with its regression-suite resolver, the NBVE shadow window for changes that touch the runtime behavior of an active module, and the SPC gate that turns shadow observations into a statistical promotion decision. The tier verifier is the gate that says the proposal is in-scope for the tier it claimed. The triad is the gate that says the proposal is in-fact better than what is currently running.
