# Chapter 18: Sandbox, NBVE, and Shadow Evaluation

A proposal that has passed the tier verifier has earned only one piece of evidence: its declared scope is admissible at its claimed tier under the governing policy. The verifier has not asked whether the proposal resolves the failure that motivated it, whether the proposal breaks any behavior that worked before, or whether the proposal performs differently against live traffic than the rule it would replace. Those are three distinct questions, and they require three distinct kinds of evidence. The first is answered by the sandbox: a deterministic test environment in which the proposed mutation is applied to a copy of the affected components, the captured failure case is replayed against the modified components, and a regression suite of historical cases is replayed alongside it. The second is answered by the sandbox's regression suite specifically: the suite includes recent successful production workflows, automatically captured from the execution log, so that any regression in production-shaped traffic surfaces before the mutation reaches production. The third is answered by NBVE shadow evaluation: the proposed rule runs dual to the production rule on live inputs for a declared window, the outputs are compared event by event, and a statistical process control gate decides whether the proposal's behavior is close enough to production behavior to promote. The three stages together — sandbox, shadow, SPC — constitute the verification triad that runs between proposal and deployment for any mutation that the policy does not auto-deploy on sandbox alone. Each stage addresses a failure mode the others miss, and the combination is what makes safe rule evolution possible in a production system.

The sandbox executor is the first stage. The compiler generates the executor as part of `mutations.ts`, the same generator file that emits the proposal table and the tier verifier. When a proposal arrives at the sandbox stage, the executor spins up a sandbox copy of the affected expert-system components — the modules, rules, scores, and patterns named by the proposal's target set — and applies the proposed mutation to the sandbox. The application is purely deterministic: the executor parses the proposal's mutation content, validates it against the same DSL constraints the original declaration satisfied, and produces a sandbox configuration in which the candidate rule has replaced the production rule. Nothing outside the sandbox is touched. The executor then runs two replays in sequence. First, it replays the captured failure case — the input that pulled the andon cord, with its full context — against the sandbox. If the mutation does not resolve the failure case, the proposal is rejected with status `rejected_by_sandbox` and the rejection is written to the ledger; the AI's proposal did not address the problem it was proposed to solve, and no further verification is warranted. Second, if the failure case resolves, the executor runs the regression suite against the sandbox.

The regression suite is what makes the sandbox useful as a defense against unintended breakage. A naive sandbox would run only the captured failure case — does the mutation resolve the symptom? — and would catch nothing about what the mutation does to inputs the AI did not see. The Agicore sandbox runs a regression suite assembled from two sources. The first source is the golden set: hand-curated test cases declared in the `.agi` source as part of the affected components' specifications. Golden tests are the deliberate, human-authored cases the architect wanted to make sure the system handles correctly — boundary cases, regression cases for past bugs, representative happy paths. The second source is recent successful production workflows, captured automatically from the execution log. The regression-suite resolver is the runtime helper that turns a suite name into a concrete set of test cases. When the policy declares `REGRESSION_SUITE 24h_recent_workflows`, the resolver scans the activity log for the last 24 hours of successful workflow executions that touched any of the proposal's target components, extracts the inputs from each execution, and constructs a test case for each. When the policy declares `REGRESSION_SUITE 7d_recent_workflows`, the resolver does the same with a seven-day window. The result is a regression suite whose size and shape match the actual production load, not the abstract test cases the architect wrote at design time. A mutation that breaks production-shaped traffic surfaces in the regression suite before it reaches production, because the regression suite is production-shaped traffic.

The executor runs the full regression suite against the sandbox and records test evidence on the proposal row: which cases were replayed, which passed, which failed, the aggregate regression metric value, coverage statistics, and a short summary suitable for inclusion in the ledger entry. The evidence is structured and queryable. An audit record might read "Mutation deployed with auto-approval per `classification_policy` T2 rules. Test evidence: one case resolved (alert_id 4729), twelve thousand eight hundred forty-seven regression cases unchanged, zero regressions detected." The evidence is not a summary the architect drafted; it is the literal output of the sandbox executor, hashed into the ledger entry, recoverable on inspection. A proposal whose regression suite has any failure is, by default, rejected with `rejected_by_sandbox` and is not auto-deployed; the policy can override this with a tolerance threshold for tiers where a small regression rate is acceptable, but the default is zero. The sandbox is the cheap-and-fast gate that catches obvious regressions without consuming live traffic.

The sandbox is sufficient evidence for tiers whose mutation class is bounded in blast radius — T1 threshold tuning, T2 rule addition. The HOC policy from the previous chapter auto-deploys T1 threshold-tuning mutations after the sandbox passes, with the 24-hour regression suite as the mechanical guarantee that nothing operational has been broken. T2 mutations might additionally require a monitoring window after deployment — the `MONITORING_WINDOW` field declares the duration over which post-deploy telemetry is watched for anomalies — but the deployment itself happens on the sandbox pass. For mutation classes whose blast radius is broader — T3 rule modification, which changes the behavior of a rule that has been firing in production — the sandbox alone is not sufficient. The mutation might pass every regression case the resolver assembled and still degrade behavior on inputs the suite did not cover. The next stage of the triad addresses exactly this gap.

NBVE — Non-Blocking Variant Evaluation — is the second stage. The primitive was introduced in Chapter 14 as the substrate for comparing model outputs in cost-optimization scenarios, where a candidate model runs in shadow against a production model to determine whether the candidate's quality is acceptable before promoting it. The Andon Loop extends the primitive from model variants to rule and workflow variants: the candidate rule runs in shadow against the production rule on live inputs, the outputs are compared, and the comparison feeds a promotion decision. The mechanics are the same; the participants differ. The integration point is a single TypeScript helper that the compiler emits and that the application code calls at every site where a candidate rule might shadow production. The helper has a precise signature.

```typescript
function evaluateWithShadow<TIn, TOut>(
  target: string,
  input: TIn,
  prodFn: (input: TIn) => TOut,
  shadowFn: (input: TIn, mutationContent: string) => TOut,
  options?: { equals?: (a: TOut, b: TOut) => boolean }
): TOut;
```

The helper is parameterized by the input and output types of the rule it wraps. The application code at the call site looks like the example below, taken from a classification workflow that uses the helper to shadow-test candidate rule modifications without instrumenting the call site twice.

```typescript
function classifyAlert(alert: Alert): Classification {
  return evaluateWithShadow(
    'alert_classification',
    alert,
    (input) => productionClassifier(input),
    (input, mutationContent) => applyMutationAndClassify(input, mutationContent),
  );
}
```

The semantics of the helper are deliberately simple. Production always runs first. The helper invokes `prodFn(input)` and the result is what the caller receives. If no proposal is in `shadow_evaluating` status targeting `alert_classification`, the helper returns the production result and incurs zero additional overhead — no shadow function is invoked, no observation is recorded, no comparison happens. The shadow path is dormant by default. When a proposal is in `shadow_evaluating` status targeting the named site, the helper invokes `shadowFn(input, mutationContent)` after the production call returns, passing the proposed mutation's content so the shadow function can apply it. The shadow function executes in the same process, on the same input, with the candidate rule applied. The helper compares the two outputs using JSON deep-equal by default, or a custom `equals` function passed in `options` for cases where structural equality is not the right comparison. The comparison result — match or differ — is recorded as a shadow observation via the `recordShadowObservation` helper, tagged with the proposal id, the production output, the shadow output, the comparison result, and a timestamp. The caller receives the production result regardless. The shadow output is never returned to the caller, never emitted to downstream systems, never visible to users.

The design properties of the helper matter. The default-dormant path means the helper can be inserted at every relevant call site at low cost, and shadow evaluation is paid for only during active windows. The production-first ordering means the user-visible behavior is unaffected by shadow execution — the production rule's output is what the application emits. The custom-equals option means structural equality is not assumed where it does not apply; a classifier whose output includes a model-generated explanation field can compare the structured classification while ignoring the natural-language explanation. The recording is mechanical: every observation is written to the `mutation_shadow_observations` table, indexed by proposal id and timestamp, and the table accumulates evidence over the window declared in the policy. A proposal in `shadow_evaluating` status for the seven days the policy specifies accumulates one observation per call site invocation over that window — potentially thousands or millions of observations, depending on the call site's traffic.

The third stage is the SPC promotion gate. SPC — statistical process control — was developed in Chapter 10 of the companion CSE textbook as the methodology for distinguishing common-cause variation from special-cause variation in a production system. The Andon Loop applies SPC to the shadow observations: given a sample of comparisons between production output and shadow output, is the defect rate (the fraction of comparisons where shadow differs from production) low enough, and is the sample size large enough, to support the conclusion that the shadow rule's behavior is acceptable for promotion? The gate is expressed as a conjunction.

```
defect_rate ≤ threshold  AND  samples ≥ min
```

The threshold and minimum are policy parameters; conventional values are a defect-rate threshold of one percent and a sample minimum that depends on the call site's traffic — a few hundred observations for low-traffic sites, several thousand for high-traffic sites. The gate logic, generated as part of `shadow_eval.ts`, runs in three steps. First, count the observations in the window: total observations, observations where shadow matched production, observations where shadow differed. Second, compute the defect rate as the ratio of differing observations to total observations. Third, compare against the policy's threshold and minimum. If defect rate is below the threshold and the sample count exceeds the minimum, the gate passes and the proposal transitions to `deployed`. If the defect rate exceeds the threshold, the gate fails and the proposal transitions to `rejected` with the test evidence preserved in the ledger entry. If the sample count is below the minimum — the window elapsed but the call site did not receive enough traffic for a confident decision — the proposal transitions to `escalated`, kicking the decision to a human approver who can either extend the window, force promotion despite the limited sample, or reject the proposal pending more evidence.

The state transitions of a proposal moving through the triad are worth enumerating.

```
proposed
  -> tier_verified (verifier passes) | tier_rejected (verifier fails, terminal)

tier_verified
  -> tested (sandbox passes against failure case and regression suite)
   | rejected_by_sandbox (sandbox fails, terminal)

tested
  -> shadow_evaluating (NBVE window required)
   | deployed (auto-deploy enabled, no shadow window)
   | escalated (approval required, no shadow window)

shadow_evaluating
  -> deployed       (window elapsed, SPC gate passes)
   | rejected       (window elapsed, SPC gate fails, terminal)
   | escalated      (window elapsed, insufficient samples)
```

The transitions are mechanical. Each is written to the ledger as it occurs. The proposal row carries a status field that reflects the current stage; the ledger carries the full history of transitions with hash-chained integrity. The terminal states — `tier_rejected`, `rejected_by_sandbox`, `rejected` — are dead ends: a proposal in a terminal state cannot transition further, but its record persists for audit. The ledger is the system's memory of every proposal that ever was, including the ones that never reached production.

The cadence at which proposals move through the shadow stage is governed by a background poller. The poller is a TypeScript function the compiler emits as `start_shadow_finalizer`, registered with the Tauri runtime at application startup. It wakes every sixty seconds, queries the `mutation_shadow_evaluations` table for rows whose window has elapsed but whose status is still `shadow_evaluating`, runs the SPC analysis for each such row, and writes the resulting transition to the ledger. The sixty-second cadence is a compromise: short enough that decisions are timely, long enough that the poller's load is negligible. A proposal whose window elapses at 14:03:27 will be finalized by the next poller tick — at the latest by 14:04:27 — and the transition will be visible in the ledger and in the mutation console within seconds of being computed. The poller is the runtime closer that turns the shadow window from a declarative duration into an executable lifecycle stage. Without it, proposals would accumulate in `shadow_evaluating` forever; with it, every shadow window ends in a deterministic verdict.

The interaction between the triad's three stages and the policy's tier definitions produces a small matrix of operational patterns. T1 threshold tuning typically auto-deploys on sandbox pass with no shadow window. T2 rule addition typically auto-deploys on sandbox pass with a post-deploy monitoring window for observation but no pre-deploy shadow. T3 rule modification typically requires a shadow window — seven days is the conventional value — with SPC promotion. T4 structural changes typically require both a shadow window and human approval, with approval gating the transition from `shadow_evaluating` to `deployed`. T5 architectural changes typically require approval at the front of the lifecycle, before any test resources are consumed. The matrix is not enforced by the compiler; it is the operational convention the field has settled on. The compiler enforces only the structural constraint — if the policy declares a shadow window for a tier, every proposal at that tier goes through the shadow stage; if the policy declares an approval authority, every proposal at that tier requires the corresponding signatures.

The worked example below traces a single proposal through the full triad. The proposal originates not from an andon pull but from the proactive `IMPROVEMENT_REASONER` declared by the policy: a weekly REASONER that scans the prior seven days of activity log against the system's `SUCCESS_METRIC`, identifies rules whose firing patterns suggest consolidation or threshold adjustment, and proposes efficiency mutations. The REASONER notices that the `urgency_classifier` rule on the `customer_support` module has been firing on inputs that the human escalation queue has consistently re-classified as routine, suggesting the rule's urgency threshold is too low. It proposes a T1 threshold-tuning mutation: raise the threshold from 0.62 to 0.68. The proposal enters the lifecycle with `claimed_tier = 1` and `claimed_scope = [SCORE_THRESHOLD]`.

The tier verifier checks the claim. The policy admits `SCORE_THRESHOLD` in T1; resolved tier is one; claimed tier is one; the proposal transitions to `tier_verified`. The sandbox executor applies the proposed threshold to a sandbox copy of the `customer_support` module, replays the captured cases that motivated the proposal — the seventeen recent inputs whose original classifications were overridden by human reviewers — and observes that the new threshold reclassifies fifteen of the seventeen as routine, matching the human verdict. The executor then runs the 24-hour regression suite of recent successful classifications, totalling twelve thousand four hundred three cases, and observes that the new threshold changes the classification on four of them — three from urgent to routine (which were borderline cases the team has long debated) and one from routine to urgent (which was an edge case the team confirms is correctly urgent under both thresholds). The regression rate is four out of twelve thousand four hundred three, well below the policy's tolerance, and the executor records the test evidence and transitions the proposal to `tested`.

Because the policy declares a seven-day NBVE window for T3 rule modifications but only a sandbox gate for T1 threshold tuning, the proposal would normally auto-deploy at this point. But the policy's author has chosen, for this specific module, to require a shadow window on T1 mutations as well — the customer-support module is operationally sensitive and the team wants shadow evidence before any threshold change reaches production. The proposal transitions to `shadow_evaluating` and the `evaluateWithShadow` helper at the classification call site begins dual-executing the production threshold and the candidate threshold on every incoming alert. Over the seven-day window, the call site receives forty-two thousand one hundred eighteen alerts. The helper records an observation for each. The SPC gate's threshold is one percent and its sample minimum is one thousand.

When the window elapses, the poller wakes, runs the SPC analysis, and reports a defect rate of zero point four percent — one hundred sixty-eight observations where shadow differed from production, out of forty-two thousand one hundred eighteen total. The sample count is well above the minimum. The defect rate is well below the threshold. The gate passes. The proposal transitions to `deployed`, the new threshold replaces the production threshold, the ledger records the transition with the test evidence and the shadow statistics, and the next alert that arrives at the call site is classified by the new rule. The improvement reasoner's proposal — a small threshold change motivated by a pattern in the activity log, validated by the sandbox against historical traffic, confirmed by shadow execution against live traffic, audited end to end — has reached production. The total elapsed time was seven days plus the few seconds of poller cadence. The total human involvement was zero. The total operational burden, beyond the architect's original `.agi` declaration, was none.

The triad's three stages address failure modes the others miss. The sandbox is fast and cheap and catches obvious regressions against historical traffic; it does not catch subtle degradations against live traffic the historical regression suite did not represent. The shadow window is slow and accumulates observations from live traffic; it does not catch failures that happen below the call site or that affect inputs the shadow path does not exercise. The SPC gate is statistical and converts shadow observations into a verdict; it does not catch the cases the comparison function did not consider a defect. Each stage's failure mode is the other stages' load-bearing case. The sandbox catches the rule that breaks the regression suite; the shadow catches the rule that passes the regression suite but degrades on live traffic; the SPC gate catches the rule whose degradation is statistical rather than categorical. The three together are what makes the Andon Loop's claim — that AI can safely propose changes to a running deterministic system — operationally credible.

The next chapter develops the human side of the lifecycle for proposals that the triad has cleared but the policy does not auto-deploy. Approval chains in three forms, the SHA-256 hash-chained ledger that records every transition with cryptographic integrity, and the file-system mirror that turns the ledger into a regulatory-submission artifact. The triad is the gate that says the proposal is in-fact better than what is currently running. The approval chain is the gate that says a human with the right authority has agreed it should run. The ledger is the gate that says, forever, that the lifecycle proceeded exactly as the architecture promised.
