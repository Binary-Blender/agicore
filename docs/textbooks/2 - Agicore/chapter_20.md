# Chapter 20: Andon Responders and Improvement Reasoners

Every gate in the Andon Loop — the tier verifier, the sandbox, the SPC-bounded shadow window, the approval chain, the hash-chained ledger — exists to govern changes once they have been proposed. None of those gates originate changes. Something has to produce the mutation in the first place. In Agicore that something is one of two AI components, declared in the DSL and emitted as concrete TypeScript orchestrations around the framework's `send_chat` interface. The two components mirror two distinct postures the system takes toward its own behavior. The first, `ANDON_RESPONDER`, is reactive: it fires when the deterministic runtime encounters a condition it cannot handle and pulls the andon cord. The second, `IMPROVEMENT_REASONER`, is proactive: it runs on a declared schedule, reviews recent execution against the success metric, and proposes refinements even when nothing has visibly broken. This chapter treats both, the memory structure that makes them safe to invoke repeatedly, and the principle from Cognition Systems Engineering that they realize in declarative form.

The reactive path begins at the boundary where determinism fails. The deterministic runtime — rules, workflows, modules, scores, patterns, actions — is built to handle the cases the DSL author has anticipated. When the runtime encounters a case outside that envelope, it does not improvise. It pulls the andon cord. The cord is a typed event placed on a named channel, carrying enough context for a downstream consumer to understand what happened, where, with what inputs, and against what expected behavior. The seven trigger conditions named in Chapter 16 are the complete set of ways the runtime can pull the cord:

| Trigger | When |
|---|---|
| `RULE no_match` | A module declared `EXPECTS_MATCH true` evaluated against an input and no rule's CONDITIONS matched |
| `STAGES_transition_guard_failed` | An entity attempted a state transition but the guard expression evaluated false |
| `ACTION_error_variant` | An ACTION's typed output union resolved to the `error` variant rather than a success variant |
| `SCORE_threshold_breach` | A SCORE crossed a threshold whose THEN clause was `ANDON` rather than a corrective action |
| `PATTERN_no_capture` | A PATTERN declared `REQUIRED true` was evaluated against text and produced no capture |
| `WORKFLOW_step_timeout` | A WORKFLOW STEP exceeded the duration declared in its TIMEOUT clause |
| `EXTERNAL_response_unparseable` | An incoming EDI message, webhook payload, or API response failed to validate against its PACKET schema |

Each of these seven is a different shape of "I cannot proceed." A rule no-match means the classifier exhausted its rule set without a verdict. A guard failure means the state machine refused a transition that something or someone attempted. An error variant means an action ran and returned an explicit failure rather than a result. A score breach means an accumulating metric crossed a line that the author declared to be intolerable. A pattern miss means a structured extractor failed where the author had promised a capture would exist. A step timeout means a workflow's externally-imposed deadline ran out. An unparseable response means an external system spoke in a dialect the system did not anticipate. The runtime catches each of these structurally — not by inference, not by anomaly detection, but by direct observation that a declared expectation was violated.

When the cord is pulled, the runtime persists an `AndonEvent` record into the database and emits an `AndonStateChange` packet onto the channel the bound `MUTATION_POLICY` declares as its `ANDON_RESPONDER` invocation source. The persistence is not optional. The record carries the event's full context: which trigger fired, which module or rule or workflow originated it, the input that produced the failure, the activity-log entries leading up to it, the timestamp, the workflow run identifier. The record is also linked, when the responder produces a proposal, to the mutation it generates. This linkage is the substrate of mutation memory, which the chapter returns to shortly.

The `ANDON_RESPONDER` itself is declared as a `REASONER` with a specific shape. The Home Operations Center's reference implementation declares it this way:

```agi
REASONER andon_handler {
  DESCRIPTION  "When the home network pulls an andon cord, diagnose and propose a mutation"
  INPUT        { CHANNEL probe_events  WINDOW "1d" }
  OUTPUT       { PACKET AndonStateChange }
  SCHEDULE     on_demand
}
```

The `SCHEDULE on_demand` is the marker that distinguishes a responder from an improvement reasoner. Where an improvement reasoner declares a cron-style or interval schedule, a responder declares `on_demand` and is invoked by the runtime whenever an andon event arrives on its bound channel. The codegen wires the binding: the generated `responder.ts` emits a `respond_to_andon` Tauri command that the runtime calls with the event identifier, and a TypeScript orchestration `respondToAndonWithAI` that loads the event's context, packages it into a prompt that includes the recent activity log and the prior mutation history for similar events, calls `send_chat` against the AI service declared in the `MUTATION_POLICY`'s scope, parses the model's JSON response into a `MutationProposal`, and submits the proposal to the same `create_mutation_proposal` command that any other source would use.

The responder is not a special-cased AI invocation. Once its proposal is created, it joins the queue. The tier verifier evaluates it, the sandbox runs it against the captured failure case and the regression suite, the optional shadow window observes it, and the approval chain — if the resolved tier requires one — collects the necessary signatures. The responder's AI involvement ends at proposal creation. Everything after that point is mechanical. This boundary is what makes the reactive path safe to invoke under any operational pressure: the worst the AI can do is propose a mutation that the deterministic gates will reject.

The proactive path is structurally similar but inverts the trigger. The `IMPROVEMENT_REASONER` runs on a declared schedule rather than on demand. The Home Operations Center's reference declares its improvement reasoner this way:

```agi
REASONER weekly_kaizen {
  DESCRIPTION  "Weekly review of network behavior; propose rule refinements"
  INPUT        { CHANNEL device_observations  WINDOW "7d" }
  OUTPUT       { PACKET AndonStateChange }
  SCHEDULE     weekly
}
```

The `SCHEDULE weekly` is the load-bearing detail. The generated `improver.ts` wires a `start_improvement_scheduler` background process that wakes hourly, checks every declared improvement reasoner's schedule against the time of its last invocation, and spawns the reasoner when its cadence has elapsed. The orchestration `runImprovementCycleWithAI` reads the activity log for the declared `WINDOW` (`7d`, `30d`, or a custom duration), computes the success metric defined in the bound `MUTATION_POLICY`'s scope, identifies efficiency candidates, packages them into a prompt, calls `send_chat`, and submits any returned mutations through the same proposal pipeline as the responder.

The notion of an "efficiency candidate" is what gives the improvement reasoner its discipline. The reasoner does not propose mutations on a whim or because the model has an opinion. It looks for four specific patterns in the activity log, and it proposes mutations only when one or more of those patterns is detected. The four kinds of candidate are:

The first kind is **steps that execute frequently without affecting downstream behavior**. The reasoner identifies workflow steps whose outputs are not consumed by any later step in the same workflow, or whose outputs are consumed but never differ across runs in a way that changes the run's outcome. Such steps are candidates for removal. The mutation might delete the step entirely or replace it with a cached lookup. The success metric must improve — or at least not regress — for the proposal to survive the sandbox.

The second kind is **sequences that exceed the metric's target**. If the success metric includes a latency or cost component and a particular workflow path consistently overruns its target, the reasoner identifies the slowest step or the most expensive sub-sequence and proposes a restructuring: parallelize two serial steps that have no data dependency, batch three sequential calls into one, replace a remote call with a local computation. Each proposal must show, in the sandbox, that the new structure matches the old structure's output on the regression suite while improving the metric.

The third kind is **rules that fire often with low subsequent action diversity**. If five rules in a module fire ten thousand times a week but all five trigger the same single downstream action with no further branching, the rules are candidates for consolidation. The proposal merges the rules into one whose CONDITIONS is the disjunction of the original five, preserving the firing behavior while reducing the rule-set surface that humans must reason about. The deterministic test catches any case where the disjunction admits a vector that any of the original five would have excluded.

The fourth kind is **workflows whose outcomes vary in ways the success metric does not capture**. If a workflow's outputs cluster into two or three distinct distributions, and the metric currently averages over the clusters, the cluster structure is invisible to the optimization loop. The reasoner proposes a new SCORE that captures the cluster — perhaps a binary score that flags one cluster as anomalous — so that subsequent improvement cycles can act on the distinction. This kind of proposal is itself a structural change, typically T4, and reaches production only after human approval.

The interaction between the two reasoner types and the rest of the loop is asymmetric in a way worth naming. Andon-response mutations frequently target T1 and T2 — threshold adjustments and new classifiers — where the auto-deploy path is open and a monitoring window catches anything the sandbox missed. Improvement mutations more often target T3 and above, because consolidation and restructuring touch existing rules and workflows. T3 mutations always go through NBVE shadow evaluation; T4 and T5 mutations additionally require human approval. The responder's instinct is to fix what just broke; the improvement reasoner's instinct is to refactor what has been working. Both pass through the same gates, but the gates apply with different weight to the two kinds of change.

The mutation memory is the second component that makes the reactive path robust. Without memory, an andon responder presented with a recurring failure pattern would propose the same kind of fix every time, regardless of whether earlier instances of that fix succeeded or failed. The MutationLedger — already the audit trail described in Chapter 19 — serves as the responder's persistent memory. The orchestration `respondToAndonWithAI` queries the ledger for prior proposals that targeted the same rule, the same workflow, or the same andon trigger category, and includes a summary of those prior proposals in the prompt context. The summary includes each prior proposal's diff, its sandbox test result, its shadow-evaluation outcome if any, its final disposition (deployed, rejected by sandbox, rejected by authority, rolled back), and any human approver notes.

The enforcement is mechanical. When the responder produces a proposal, the create-proposal path performs a pre-acceptance check: does the new proposal's payload match a previously-rejected proposal targeting the same failure pattern within a configurable lookback window? If it does, the proposal is auto-rejected without test, with the rejection reason `same_pattern_same_fix_already_rejected`. The check is exact-match on the canonical-JSON serialization of the mutation payload, scoped to proposals whose linked andon event shares the same trigger category and target. This prevents the responder from looping: the AI cannot waste sandbox cycles proposing the same fix that humans or the SPC gate already declined.

The complement to that check is the meta-andon escalation. When the same failure pattern recurs *despite* a previously-deployed fix, the system recognizes that the deployed fix did not hold. The recurring failure produces a new andon event linked to the prior fix's proposal via the `addressed_by_proposal_id` field. The responder, on its next invocation, sees the linkage and produces a proposal flagged as `meta_andon: true`. The create-proposal path bumps the resolved tier upward — a T1 fix whose target is now meta-andoning is re-tiered to T4, which forces human approval and a structural review. The escalation is automatic, not optional. The pattern recognizes that a fix that fails to hold is no longer a fix at the original tier's scope; it is an architectural symptom that the original analysis missed something structural, and the structural tier is where it belongs.

The reactive/proactive decomposition deserves to be defended on its own terms, because the obvious alternative — a single AI loop that watches for both failures and inefficiencies — is the one most "agentic" systems converge toward, and it produces worse results. A single loop cannot serve both purposes well because the timescales and the context shapes disagree. A failure demands fast response with narrow context (this specific event, this specific failure case, these specific prior fix attempts); an improvement demands slow analysis with wide context (the last week of activity, the full distribution of metric values, the cross-cutting patterns that no single event reveals). Conflating the two produces an AI loop that is slow to respond to failures because it is busy doing wide analysis, and shallow in its improvements because it is reacting to recent events at improvement cadence. The reactive/proactive split assigns each posture to its own component, each with its own input shape, its own schedule, its own prompt, and its own success criterion. The shared infrastructure — the proposal pipeline, the gates, the ledger — is what they have in common; the rest is deliberately separated.

Both reasoners produce, structurally, the same artifact: an `AndonStateChange` packet that becomes a `MutationProposal` row. The packet carries the proposed diff, the claimed tier, the rationale text from the AI, and the linkage to the originating andon event or improvement-cycle record. From the perspective of every gate downstream of proposal creation — the tier verifier, the sandbox, the shadow evaluator, the approval chain, the ledger — there is no distinction between a proposal that originated reactively and one that originated proactively. The gates treat them uniformly. This uniformity is intentional. It means that a single set of mechanical guarantees applies to every change the system makes to itself, regardless of which AI posture produced it. There is no "fast path" that bypasses verification because a fix was urgent; there is no "trusted path" that bypasses tier checking because an improvement was scheduled by a known reasoner. Every proposal runs the full gauntlet.

The realization of this architecture in the Home Operations Center deserves a worked traversal because it grounds the abstractions in operational reality. HOC monitors a home network, runs probe workflows against the internet uplink, scans for devices, and classifies what it finds. Two modules — `internet_health` and `device_classifier` — each declare `EXPECTS_MATCH true` and bind to a `MUTATION_POLICY` named `network_policy`. The policy declares three tiers (threshold tuning at T1, new classifier at T2, governance at T5), the `andon_handler` reasoner as the responder, the `weekly_kaizen` reasoner as the improvement loop, and the `network_ledger` as the audit destination.

A `RULE no_match` event arrives on the andon channel when the `device_classifier` module encounters a new MAC OUI prefix it has never seen. The responder fires. It reads the failure case (MAC `aa:bb:cc:dd:ee:ff`, observed making mDNS queries on port 5353), the activity log for the last day, and the mutation history for prior classifier additions. It calls `send_chat`, asking the model to propose a classification rule. The model returns a proposal: a new rule `classify_apple_homepod` whose CONDITIONS check the OUI prefix against Apple's published assignments and whose ACTIONS set the device type to `smart_speaker`. The proposal claims `MUTATION_TIER 2`. The tier verifier accepts (`RULES_modify` is in the T2 scope). The sandbox runs the proposal: the captured failure case now classifies, and 4,200 regression cases across the prior 24 hours of network observations classify identically. The proposal moves to `TESTED`. Because T2 declares `AUTO_DEPLOY false` and `APPROVAL_AUTHORITY chris`, the proposal enters the approval queue. Chris signs, the proposal deploys, and the ledger records the full nine-step lifecycle.

A week later, the `weekly_kaizen` reasoner fires on its schedule. It reads the device-observation channel for the prior seven days and computes the success metric — a composite of classification coverage and false-classification rate. It identifies three rules in `device_classifier` that fired on overlapping conditions and consistently triggered the same downstream action. It proposes a consolidation: merge `classify_smart_bulb_lifx`, `classify_smart_bulb_philips`, and `classify_smart_bulb_govee` into a single `classify_smart_bulb` rule whose CONDITIONS disjoin the three vendors' OUI prefixes. The proposal claims `MUTATION_TIER 3` (existing rule modification). The verifier accepts. The sandbox runs the consolidated rule against 7 days of observations and confirms identical classifications to the three original rules. The proposal enters NBVE shadow evaluation for 7 days; the SPC gate measures defect_rate against live observations and finds zero divergence; the SPC gate promotes; the policy at T3 does not require human approval; the consolidated rule deploys; the ledger records each transition.

These two traversals are, end to end, what the Andon Loop produces in normal operation. The reactive path closes the loop on a novel device by proposing a new classifier that humans approve. The proactive path closes the loop on classifier sprawl by proposing a consolidation that shadow testing validates without human intervention. Both produce immutable ledger entries. Neither requires the AI to be in the runtime decision path. The deterministic system continues to classify devices on every probe; the AI's involvement is bounded to edit-time proposal creation, and the proposals reach production only through the same set of mechanical gates.

This is the Cattle Dog Principle in declarative form. Chapter 7 of *Cognition Systems Engineering* names the principle and develops it in detail: in a well-designed cognitive system, the AI plays the role of the working dog and the human plays the role of the shepherd. The dog is fast, tireless, capable of covering ground the shepherd cannot. The shepherd holds final authority over consequential decisions — where the flock goes, when it moves, which sheep are sold, which are kept. The dog's job is not to make those decisions; the dog's job is to provide capability and throughput within the shepherd's directed framework. The principle generalizes: AI provides interpretation, generation, and analytical breadth; humans hold authority over consequential change. The Andon Loop's declarations realize this principle mechanically. `ANDON_RESPONDER` and `IMPROVEMENT_REASONER` are the AI components — the working dogs, fast to react, tireless in their analysis. `MUTATION_POLICY` with its `APPROVAL_AUTHORITY` clauses is the shepherd's authority — explicit, named, positionally enforced. The tier verifier, the sandbox, and the SPC gate are the fences that ensure the dogs cannot mistakenly drive the flock off a cliff. The hash-chained ledger is the record by which the shepherd can prove, after the fact, exactly what the dogs did and exactly what authority sanctioned each consequential movement.

The principle's force comes from the asymmetry it codifies: AI's strengths and AI's weaknesses are not symmetric, and a system that respects the asymmetry produces better outcomes than one that treats AI as a uniform resource to be deployed wherever it might help. AI's strengths — interpretation of novel inputs, generation of candidate solutions, analytical recognition of patterns across large activity logs — are exactly the strengths that proposal-time work requires. AI's weaknesses — stochasticity, susceptibility to adversarial inputs, audit opacity, the inability to mechanically guarantee that a future invocation will produce the same result — are exactly the weaknesses that runtime execution cannot tolerate. The Andon Loop's two AI components live where the strengths pay and the weaknesses are bounded. The deterministic runtime executes where the weaknesses would compound and the strengths are not required.

The mapping is not metaphorical decoration. Each piece of the principle has a corresponding piece of the declaration surface. The dog's enthusiasm — the AI's eagerness to propose — maps to the responder and improver invocations. The dog's training boundaries — what the dog has been taught it may and may not do — map to the SCOPE and MUTATION_TIER fields. The shepherd's whistle commands — the explicit human authorizations — map to the approval chain forms. The shepherd's accountability to the landowner — the auditor's eventual right to know what happened — maps to the ledger. A declarative system that omits any of these pieces is implementing the principle incompletely; a declarative system that includes all of them is implementing it precisely.

Part IV closes here. The Andon Loop is the architecture that lets AI participate in the evolution of a deterministic system without compromising the determinism that makes the runtime trustworthy. The inversion is structural: AI lives at the edit boundary, proposing changes; the deterministic expert system runs the rules; every proposal passes through mechanical gates that AI cannot bypass; every consequential change is signed by the humans the policy names; every transition is recorded in a tamper-evident chain. The five chapters of this part have walked the architecture from its conceptual inversion through its policy syntax, its sandbox and shadow evaluation, its approval and audit mechanisms, and its two AI entry points. What remains is to see the architecture at work. Part V turns to the reference implementations — the canary, the home operations center, the production ops platform, the eighteen-application enterprise stack — that demonstrate everything Part IV has described in systems that run, in production, today.
