# Chapter 5: From .agi to Tauri — The Compilation Pipeline

A compiler is the place where a discipline accumulates. The grammar is a snapshot of which constructs the community considers primitive. The static checks are a snapshot of which mistakes the community has decided are not worth tolerating. The generators are a snapshot of which patterns the community has decided are no longer worth writing by hand. Read in this register, the Agicore compiler is not infrastructure beneath the architecture — it is the architecture, frozen into a transformation. Every claim Part I made about the inversion, about runtime determinism, about the DSL as constraint boundary, cashes out in roughly nine thousand seven hundred lines of TypeScript that sit in `core/compiler/src/generators/` and run in well under a second on a laptop.

This chapter walks one declaration end-to-end. A single line of `.agi` source becomes a SQLite migration, a Rust struct, a Tauri command, a TypeScript interface, a typed invoke wrapper, a slice in a Zustand store, a list view, and a form modal. The walk is mechanical. Nothing in it requires AI. Nothing in it requires runtime branching. Every transformation is a function from one data structure to another, and every transformation is independently testable. The compiler's job, examined closely, is the elimination of categories of error by construction. We will trace exactly which categories disappear at which stage, and why the residual errors after `cargo build` and `tsc --noEmit` pass are application-level concerns rather than wiring concerns.

The pipeline has ten stages. Source text enters at the top; an output project tree exits at the bottom. The intermediate stages — token stream, AST, validated AST, emitted artifacts, written files, two-compiler verification — are each a stable interface. Anyone reading the compiler reads it as Knuth would have one read a literate program: not as a monolith but as a chain of small, named transformations, each defensible on its own. The first transformation is character to token. A `.agi` file is opened, its bytes are read, and a hand-written lexer in `core/parser/src/lexer.ts` scans left to right, emitting a stream of typed tokens. The lexer maintains a single keyword map; every identifier is looked up against that map; matches become keyword tokens, misses remain identifier tokens. Whitespace is consumed and discarded. Comments — both `//` line comments and the protective `// @agicore-protected` marker — are recognized at this layer because downstream stages must see them.

It is worth pausing on the choice of ten stages as the right granularity for the pipeline, because the choice is not arbitrary and the alternative — fewer, larger stages — would have produced a different compiler. A monolithic compiler that read source text and emitted output files in a single pass would be simpler to write initially and substantially harder to test, modify, and extend. The ten-stage decomposition pays a small cost in interface plumbing — each stage must produce a value the next stage consumes — and a large dividend in testability. Each stage can be exercised in isolation: feed the lexer a string, assert on the token stream; feed the parser a token stream, assert on the AST; feed a generator an AST, assert on the output file. The compiler's three thousand-test suite is dominated by tests of this shape, and the shape is only possible because each stage has a defined contract with its neighbors. Aho, Sethi, and Ullman make essentially the same argument in the dragon book's introductory chapter: the compiler-as-pipeline is the right architecture not because the stages are conceptually distinct but because the contracts between stages are the unit of correctness reasoning the team can actually maintain.

The lexer's discipline is worth noting before we move past it. It is hand-written and small, on the order of a few hundred lines. There is no parser-generator, no DSL-of-DSLs, no compiler-compiler. Aho, Sethi, and Ullman's *Compilers: Principles, Techniques, and Tools* devotes its early chapters to the case for lexer generators; the Agicore implementation rejects that case for a specific reason. The keyword set is the discipline. Adding a keyword is a decision about what the field considers primitive. That decision should be made by reading and editing the keyword map directly, not by editing a grammar file that compiles into something that edits the keyword map. The cost of hand-maintenance is low. The cost of indirection on a load-bearing artifact is high. The compiler favors the cheap cost.

Consider the running example: `ENTITY Student { name: string }`. The lexer produces a stream that reads, approximately, `ENTITY_KW`, `IDENT("Student")`, `LBRACE`, `IDENT("name")`, `COLON`, `IDENT("string")`, `RBRACE`. Each token carries its source line and column. The lexer does not know what an ENTITY is; it knows only that the seven characters `E-N-T-I-T-Y` followed by a non-identifier character match a registered keyword. The semantic work — that `ENTITY` starts a top-level declaration, that the identifier after it is the entity's name, that the braces enclose its body — belongs to the parser.

The parser is recursive-descent, and the structure of the parser mirrors the structure of the grammar one-to-one. There is a top-level loop that reads the next token, dispatches on its kind to the matching `parseFoo` function, and accumulates the resulting AST node into the program. `parseEntity`, faced with our token stream, consumes the `ENTITY_KW`, expects an identifier and binds it as the entity name, expects an opening brace, then enters a loop that parses field declarations until it sees the closing brace. Each field declaration is itself a small parser: identifier, colon, type identifier, optional modifier keywords (`REQUIRED`, `UNIQUE`, default values, foreign-key markers). When `parseEntity` returns, the AST holds a single `EntityDecl` node with `name: "Student"` and one `Field` child whose name is `"name"` and whose type is `"string"`. The source position of every node is preserved; error messages later in the pipeline will point at the line and column the user typed.

The AST is the contract between the front end of the compiler and everything that follows. Code generators consume the AST, not the source text. Static validators consume the AST. The CLI's pretty-printer consumes the AST. Anything that needs to reason about a `.agi` program reasons about it as a tree of typed nodes whose shapes are declared in `core/parser/src/types.ts`. This is not an accident of implementation. It is the same separation of concerns that Wirth advocated when he argued that algorithms and data structures are the two complementary halves of a program and that getting the data structure right was the larger half. The AST is Agicore's data structure. Get it right, and the generators are short and obvious. Get it wrong, and the generators grow byzantine workarounds.

Between the parser and the generators sits a static validator — what the compiler-theory literature would call a semantic analysis pass. The validator walks the AST and runs twelve checks. Are there duplicate declaration names? Do all `BELONGS_TO` references point at entities that actually exist? Are all action `INPUT` and `OUTPUT` types either primitive or declared? Do workflow steps reference actions that the program defines? Are pipeline node input and output types unifiable with their neighbors? Each check is a function from AST to a list of diagnostics. Errors abort compilation. Warnings are emitted and compilation continues. The twelve checks are the residue of every category of mistake the compiler's authors have caught at least once while writing real applications. They are accumulated discipline made executable.

The twelve checks themselves are worth enumerating in summary, because they are the operational definition of *semantically valid* in the Agicore DSL. The checks include: duplicate declaration names across the program; unresolved references in `BELONGS_TO`, `HAS_MANY`, workflow steps, pipeline nodes, module member lists, and skill bindings; type mismatches between declared field types and the types the rest of the program assumes; unreachable states in `STAGES` declarations; cycles in pipeline node graphs (which are required to be acyclic); inconsistent `INPUT`/`OUTPUT` typing between adjacent pipeline nodes; missing or extra fields in `BREED` definitions; conflicting `MUTATION_POLICY` tier specifications; over-broad scope claims that violate the tier verifier's pre-runtime check; missing `SIGNED_BY` clauses on `SKILLDOC` declarations that target restricted execution environments; references to AI services that are not declared in the program. The list grows as the discipline grows. Each addition is the residue of a real bug that real code shipped, until someone decided that the bug deserved a structural prevention rather than a per-application workaround.

It is worth dwelling on what the validator gives you, because it is the place where the build-time/run-time boundary is sharpest. In a conventional dynamic system, an undeclared reference becomes a `NoMethodError` at production runtime, three weeks after a deployment, when a user clicks a button no one ever clicked in testing. In Agicore, the same undeclared reference is a validator error at the developer's terminal, a hundred milliseconds after they save the file. The error is precise — line, column, the name of the missing entity, the declaration that referenced it — and the program cannot be compiled until it is fixed. Hoare's argument in his 1969 paper that program correctness is most economically established before execution, by reasoning over text rather than by inspecting behavior, applies here exactly. Twelve semantic checks are a tiny number of inference rules. They are sufficient to catch a long tail of structural errors that would otherwise surface only when a customer triggered them.

Once the AST has been validated, the generators run. Each generator is a function from the AST to a string — the contents of one or more output files. They run in a defined order. The SQL generator emits a migration file under `migrations/`. The Rust generator emits CRUD command implementations and command-registration code under `src-tauri/src/commands/`. The TypeScript types generator emits an interface file under `src/lib/types.ts`. The invoke-wrapper generator emits typed function wrappers around the Tauri `invoke` API under `src/lib/invokes.ts`. The store generator emits Zustand store slices under `src/lib/store.ts`. The React generator emits a list view and a form modal under `src/components/`. The Tauri configuration generator emits `tauri.conf.json` and the capability ACLs that bound what the application is allowed to do.

For `ENTITY Student { name: string }` the SQL output is a single migration containing roughly the following:

```sql
CREATE TABLE IF NOT EXISTS students (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL
);
```

The plural snake-case table name, the integer primary key, the `NOT NULL` constraint, the use of `TEXT` rather than `VARCHAR(255)` — all are decisions made once in the SQL generator and propagated to every entity the compiler ever sees. There is no opportunity to spell the column wrong, because no human wrote the column. The Rust generator emits a corresponding struct in `src-tauri/src/commands/crud.rs`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Student {
    pub id: i64,
    pub name: String,
}

#[tauri::command]
pub fn create_student(name: String, state: tauri::State<DbPool>) -> Result<Student, String> {
    let conn = state.0.lock().unwrap();
    conn.execute("INSERT INTO students (name) VALUES (?1)", [&name])
        .map_err(|e| e.to_string())?;
    let id = conn.last_insert_rowid();
    Ok(Student { id, name })
}
```

The `#[derive(Serialize, Deserialize)]` is what allows Serde to move this struct across the Tauri IPC boundary without hand-written marshalling code. The `#[tauri::command]` is what registers the function as callable from the frontend. Neither was written by the developer. Both are emitted from the same `EntityDecl` node that produced the SQL.

The TypeScript types generator emits, in `src/lib/types.ts`:

```typescript
export interface Student {
  id: number;
  name: string;
}
```

The case conversion — Rust's `name: String` becoming TypeScript's `name: string`, and snake-case database columns becoming camelCase TypeScript fields when relevant — happens at this boundary, deterministically, by a rule applied in the generator. The compiler enforces the convention. The developer never has to remember which side of which boundary they are on. The invoke-wrapper generator emits:

```typescript
import { invoke } from '@tauri-apps/api/core';
import type { Student } from './types';

export async function createStudent(name: string): Promise<Student> {
  return invoke<Student>('create_student', { name });
}
```

The wrapper's argument list is derived from the Rust command's argument list. Its return type is derived from the Rust command's return type. The string `'create_student'` — the command's IPC name — is the same string used in the Tauri command registration on the Rust side. The compiler emits both. They cannot drift apart, because they come from the same AST node. The Zustand store generator emits state and actions that consume the typed wrappers. The React generator emits a `StudentList` and `StudentForm` that consume the store. Each layer is type-coherent with the layer above it, by construction.

This is the inversion of the conventional anti-pattern, where developers mis-spell IPC channel names and discover the mistake at runtime when the call returns `null`. In Agicore, the channel name appears in exactly two places — the Rust command registration and the TypeScript wrapper — and both are emitted from the same AST node. A typo is impossible. The seven classes of error catalogued in the Agicore coding standards — forgotten IPC channels, type mismatches between Rust and TypeScript, mixed case in SQL, multiple Zustand stores, magic-string command names, forgotten command registrations, forgotten migrations — all become not-mistakes-you-can-make rather than mistakes-you-must-avoid. The compiler does not make these errors merely uncommon. It makes them structurally impossible.

The output project tree is worth seeing whole. After the generators run, the destination directory contains a `migrations/` folder with a single SQL file, a `src-tauri/` folder with the Rust crate (source, `Cargo.toml`, `tauri.conf.json`, capabilities ACL), and a `src/` folder with the TypeScript application (types, invokes, store, preferences, components). At the root sit a `package.json` with pinned versions, a `vite.config.ts`, a `tsconfig.json`. Everything Tauri needs to build a 5MB native application is present and consistent. Nothing the developer must remember to add is absent.

Most of the files emitted are recreated wholesale on every regeneration. This is the right default — it means the generated code is always exactly what the current AST implies, with no drift between source and output. But there is a category of file that should not be recreated: the implementation body of a deterministic action that the developer wrote in Rust. The compiler handles this with a single mechanism, the protected-file marker. The first line of a protected file is `// @agicore-protected`. When the generator is about to emit such a file and a file with that marker already exists at the destination, it leaves the existing file alone. The contract with the developer is precise: you fill in the body; the compiler keeps the surrounding signatures aligned with the DSL, but only by regenerating other files that reference your implementation, not by overwriting you. This is the same separation that Knuth's literate-programming tradition recognized between machine-generated text and human-authored text. Agicore does not weave them in the same file; it separates them into different files governed by a marker, which is easier to reason about and easier to grep for.

Once the project tree is on disk, the compilation pipeline ends and two independent compilers take over. `cargo build` walks the Rust crate, resolves dependencies, runs the borrow checker, and produces a native binary. `tsc --noEmit` walks the TypeScript code, resolves modules, runs structural type checking, and reports any inconsistency between declared and used types. Either compiler failing means the generated code is broken; in practice, neither does, because the generators emit code that has already been written successfully thousands of times across the project's history. The two-compiler property is the subject of Chapter 8, and the precise sense in which "if it builds, it works" is true will be defended there. What matters at this point in the trace is that the Agicore compiler emits code that two of the industry's most aggressive type-checking systems then verify, and any error either of them raises is a genuine signal that the DSL — or, much less commonly, the generator — needs to change.

Several practical properties of this pipeline are worth naming because they are easy to miss while watching the trace. The pipeline is fast: a complete regeneration of NovaSyn Chat's 596-line `.agi` file into its 54+ output files takes well under a second on commodity hardware. The pipeline is reproducible: the same input produces the same output byte-for-byte, which is what makes it possible to commit generated code to a repository and audit diffs across changes. The pipeline is local: it runs entirely on the developer's machine, with no network calls, no service dependencies, no telemetry. There is no compiler service. There is a TypeScript file you can read.

The pipeline is also, by design, the locus of the discipline. When you discover a category of error the current compiler does not prevent, the response is not to write a runtime check in the application — that would be regression to the runtime-LLM mindset, where every safety property is a policy you have to maintain. The response is to add a check to the static validator, or to teach the generator to emit code that cannot exhibit the error. The error becomes structurally impossible, once, for every application the compiler will ever emit. This is the kaizen loop applied to compiler engineering, and Chapter 25 will treat its dynamics in full. For now it is enough to notice that the compiler's surface area grows monotonically as the field discovers what is genuinely primitive, and that no application built on the framework needs to be touched when the compiler grows. Regenerate, and the new discipline is enforced.

A practical note on the regeneration model is in order before we close, because the regeneration model is what turns the compiler from a one-shot scaffolder into a living architectural component. The compiler is designed to be run repeatedly against the same project. Each run produces the same outputs for the unchanged declarations, the updated outputs for the changed declarations, and the same protected files for the implementation territory the developer owns. There is no separate concept of *create new project* versus *update existing project* in the compiler's vocabulary; there is only *regenerate*, and the regeneration is idempotent. This is the same design property that distinguishes Terraform from a shell script that creates infrastructure — the script knows how to create, the declarative tool knows how to converge. The Agicore compiler converges. A change to the `.agi` source becomes the diff between the previous regeneration and the new one, which is exactly the diff the version-control system shows the team.

The convergence property has a second-order effect that is worth noting because it shapes how teams think about source control. The generated code is committed to the repository alongside the `.agi` source. This is sometimes surprising to developers accustomed to treating generated code as ephemeral build output, but it is the right choice for two reasons. First, it makes the generated output reviewable in the same pull-request workflow the team uses for any other code change — a reviewer can see precisely how a one-line `.agi` change cascades through the generated artifacts, and the audit trail is in the commit history rather than lost in build logs. Second, it allows the project to be cloned and run without the Agicore toolchain installed, which is the right property for the case studies in Part V where the running application is what people want to see, not the compiler that produced it. The generated code is real code; the developer can read it, debug into it, and inspect it without indirection. The compiler's authorship of the code is a fact about its origin, not a fact about its accessibility.

The pipeline's contract with the developer extends through the rebuild loop as well. When the developer changes the `.agi` source and reruns the compiler, the compiler does not silently overwrite the protected implementation files. It does not silently add or remove files from version control. It produces a diff that the developer reviews and commits. The diff is small for small changes and large for large changes; the size of the diff is proportional to the size of the source change, which is the property that makes regenerated diffs reviewable. The convergence is not magic. It is a discipline the compiler enforces because the alternative — non-deterministic output, churn in unrelated files, silent overwrites of human work — would make the architecture untrustworthy at the team scale.

What we have walked, then, is a translation device whose load is exactly the elimination of accidental complexity. A line of DSL becomes a file system of generated artifacts, the artifacts pass two independent compilers, and the resulting binary runs deterministically on a desktop with a 5MB footprint. The intelligence that wrote the DSL — whether a human, an AI agent, or both — is no longer in the loop when the application executes.
