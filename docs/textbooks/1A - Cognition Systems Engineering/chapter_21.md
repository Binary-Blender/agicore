Every human expert carries context. A senior attorney advising a client on a complex transaction is not starting from zero knowledge of the client's situation at each meeting. She remembers the prior deal structure, the client's risk tolerance, the regulatory constraints that surfaced in the last engagement, the counterparty's negotiating posture. This accumulated context is not the same as long-term memory in the neurological sense — it is operational context: the set of facts, constraints, preferences, and history that governs how she thinks about the current situation. Her advice is not merely an application of generic legal knowledge to the current document; it is the application of specialized legal knowledge to the current document in the context of this client, this transaction, this history. Strip away the operational context and she becomes a generic legal assistant — still capable, still trained, but not the advisor the client engaged. Most AI systems in production today are generic legal assistants in precisely this sense: technically capable but contextually impoverished. Each invocation starts from the same baseline. The system does not remember prior sessions. It does not know the client's risk tolerance. It does not recall that the last synthesis run flagged a specific regulatory concern. It is always a stranger. This is not a limitation of AI models as such — it is a limitation of **stateless architecture**: the design choice, made for simplicity, to treat each request as an independent transaction with no connection to prior transactions. SESSION and MODULE are Agicore's answer to stateless architecture. They implement what might be called **stateful cognitive operation**: the ability of a cognitive production system to configure its own behavior based on operational context, to persist relevant state across invocations, and to apply specialized expertise automatically when conditions warrant. The gap between stateless and stateful cognitive operation is the gap between a stranger and an expert.

The **SESSION** declaration is the mechanism by which a cognitive system adopts a formal operating mode. A SESSION is not a conversational session in the web application sense — it is not a handle for a sequence of HTTP requests. It is a **semantic operating mode**: a formally specified configuration of the system's behavioral profile that governs how the system approaches all inference tasks while the session is active. A SESSION declaration contains five elements. The first is the **tools list**: the set of capabilities available to the system in this mode. A research mode might activate search, retrieval, and citation tools. An editorial mode might activate style checking and format validation tools. An analysis mode might activate statistical evaluation and confidence scoring tools. The tools list is not a whitelist of allowed operations — it is a specification of which capabilities the system should actively employ in this mode, shaping how reasoners approach their tasks. The second is the **context specification**: a natural language statement of the operating assumptions, goals, and background information that govern inference in this mode. The context specification is part of the prompt context assembled for every REASONER run while this session is active. It is the system's standing brief: the background that every reasoner reads before beginning its task. The third is the **constraints list**: a set of explicit behavioral boundaries that the system must respect in this mode. Constraints are negative specifications — what the system must not do — and they are as important as the positive specifications. An analytical session might constrain the system to avoid speculative claims without confidence scores. A client-facing session might constrain the system to avoid technical jargon above a defined complexity threshold. A regulatory-review session might constrain the system to recommend human legal review for any output containing liability language. The fourth is the **memory flag**: a boolean that specifies whether insights generated during this session's activations should be persisted to semantic memory for recall in future sessions. A session with `memory: true` accumulates organizational learning across activations; a session with `memory: false` treats each activation as episodic and does not persist its outputs to the knowledge store. The fifth is implicit: a session activates at the system level in response to a FIRES SESSION action from a TRIGGER, changing the system's behavioral profile for all subsequent reasoner runs until a different session is activated or the session is explicitly deactivated.

The **semantic operating mode concept** is worth developing carefully, because it represents a formalization of something that human experts do intuitively and that AI systems have historically lacked any formal mechanism for. A skilled knowledge worker does not approach all tasks with the same cognitive configuration. When writing an executive briefing, they activate a mode characterized by precision, parsimony, audience-awareness, and careful hedging of uncertainty. When brainstorming with a product team, they activate a mode characterized by generativity, openness to provocation, low filtering, and suspension of critical judgment. When reviewing a legal document, they activate a mode characterized by exhaustive attention to edge cases, conservative interpretation, and systematic risk identification. These modes are not personality traits — they are cognitive configurations that the practitioner adopts deliberately, based on the task's demands. The practitioner who cannot modulate their cognitive mode — who approaches brainstorming with the same exhaustive risk-identification orientation they bring to legal review — is less effective in both contexts. SESSION declarations formalize these configurations as system-level artifacts. They make implicit cognitive modes explicit, named, versioned, and improvable. The practitioner who has not formalized their system's operating modes is operating from implicit configurations that vary with whichever practitioner designed the last system prompt, from which mood they designed it, and on which task they were optimizing for. The SESSION declaration converts these implicit, variable configurations into explicit, stable, testable artifacts. This is the same move that the REASONER declaration makes with respect to inference — converting craft into engineering.

```
session AnalyticalMode {
  description "Deep analysis operating mode"
  tools ["research", "synthesis", "validation"]
  context "Operating in structured analytical mode. Prioritize completeness and accuracy."
  constraints ["no speculative claims without confidence scores", "cite sources for all factual claims"]
  memory true
}
```

This declaration specifies an analytical mode that activates research, synthesis, and validation tools; establishes a standing context of structured, completeness-first reasoning; enforces two specific constraints — no unconfident speculation, mandatory source citation — and persists its session insights to semantic memory. A TRIGGER that fires this session in response to a channel message containing an `AnalysisRequest` packet type changes the system's behavioral profile for all reasoners that execute during this session's activation. The `SynthesizeAnalysis` reasoner, when it runs under `AnalyticalMode`, assembles its context with the session's context specification prepended, applies its system prompt with the session's constraints as additional operating instructions, and tags its RUN record with the active session name. The session does not change what the reasoner does — the USING, CONSIDERING, PRODUCING, and MODEL clauses remain unchanged. It changes the behavioral environment in which the reasoner does it. This is the architectural equivalent of putting a human expert into a specific cognitive mode before they begin a task: same expertise, same tools, different configuration. The difference in output quality between a well-configured session and no session is the same difference as the one between a professional who has read the brief and one who has not.

The **MODULE** declaration implements a different kind of behavioral specification: not a system-wide operating mode, but a composable, conditionally activated unit of expert knowledge. A MODULE is an **expert system bundle** — a formally specified contribution of specialized expertise that activates automatically when runtime conditions warrant it, without requiring explicit invocation. A MODULE declaration has four components. The first is the **expert domain**: a natural language description of the area of expertise this module represents. The expert domain is both a documentation field and a routing label: it tells the practitioner what the module knows and tells the runtime how to represent the module's contribution in the assembled context. The second is the **patterns list**: the specific rules, heuristics, and behaviors that constitute the module's expertise. These are concrete operational directives: "flag liability language," "apply conservative confidence calibration to novel claims," "use formal definitions for all technical terms," "include edge cases in any specification of a process." The patterns list is the module's knowledge base — the encoded expertise that the module contributes to the inference context when it is active. The third is the **ACTIVATE_WHEN condition**: a logical predicate evaluated against a **fact store** — a key-value store of runtime facts asserted by triggers, operators, reasoners, and other modules. When the ACTIVATE_WHEN condition evaluates to true, the module contributes its expert domain and patterns to the active inference context. The fourth is the **score**: an integer that specifies the module's priority in cases of conflict with other simultaneously active modules.

The **fact store and activation conditions** mechanism is a direct descendant of expert system architecture — specifically, of forward-chaining production rule systems, whose history runs from the MYCIN medical diagnosis system at Stanford in the 1970s through the knowledge engineering tradition of the 1980s and 1990s and into the constraint-based architectures of contemporary formal systems. In a production rule system, facts are assertions about the world — "the patient has fever: true," "the patient's WBC count is 14,000" — and rules are condition-action pairs that fire when their conditions are satisfied by the current fact base. MYCIN's rules encoded the diagnostic expertise of infectious disease physicians: if the organism is gram-negative and the patient is immunocompromised and the site of infection is the bloodstream, recommend gentamicin. The rules were independent of each other, composable, and auditable. A physician reviewing MYCIN's recommendations could trace each recommendation to the rules that produced it and evaluate whether each rule was clinically sound. Agicore's MODULE activation mechanism applies this architecture to cognitive system configuration: the fact store is the assertion base, the ACTIVATE_WHEN condition is the rule condition, and the module's pattern contribution is the rule action. The module fires when its condition is satisfied. Its contribution to the inference context is auditable: the practitioner can see which modules activated for a given run and evaluate whether each module's activation was appropriate.

The fact store is populated by multiple sources. TRIGGER declarations can assert facts when they fire: a trigger that fires on a high-priority request packet can assert `context.priority == 'high'` to the fact store before launching the processing reasoner. Human operators can assert facts through the system's operational interface: a compliance officer can assert `context.regulatory_review_required == true` for a batch of documents that require formal review. REASONER runs can assert facts based on their outputs: a classification reasoner that determines a document concerns legal liability can assert `context.domain == 'legal'` to the fact store. Other modules can assert facts when they activate: a risk-assessment module that activates on high-priority documents can assert `context.risk_level > 0.8` when it determines the risk threshold is exceeded. The fact store is a shared, runtime-dynamic assertion base that reflects the current operational context as understood by all components of the system. MODULE ACTIVATE_WHEN conditions query this assertion base, enabling each module to respond to the full operational context as collectively understood.

Consider two modules that might co-activate on a high-priority legal document:

```
module LegalCautionModule {
  description "Apply legal risk awareness when legal context is detected"
  expert_domain "Legal and Regulatory Risk"
  patterns ["flag liability language", "recommend legal review for high-stakes claims"]
  activate_when "context.domain == 'legal' OR context.risk_level > 0.8"
  score 90
}

module TechnicalDepthModule {
  description "Apply deep technical precision when technical context is detected"
  expert_domain "Technical Documentation"
  patterns ["use formal definitions", "include edge cases", "specify invariants"]
  activate_when "context.domain == 'technical'"
  score 70
}
```

A document that is simultaneously a technical specification with legal implications — a software contract, for example — might satisfy both activation conditions: `context.domain == 'legal'` and `context.domain == 'technical'` might both be asserted in the fact store, activating both `LegalCautionModule` and `TechnicalDepthModule`. Both modules contribute their expert domain and patterns to the inference context. The assembled context for the processing reasoner includes both the legal caution patterns — flag liability language, recommend legal review — and the technical depth patterns — use formal definitions, include edge cases, specify invariants. The reasoner synthesizes a document that is simultaneously legally cautious and technically precise. This is the expert team effect: the system behaves as if a legal expert and a technical documentation expert are both reviewing the output, because both modules' expertise is encoded in the inference context.

**Score-based priority resolution** handles the cases where two active modules give conflicting directives. If `LegalCautionModule` specifies that all claims about system behavior should be qualified with appropriate uncertainty language, and `TechnicalDepthModule` specifies that all behavioral claims should be stated with precise, unhedged definitions, the modules conflict on the question of whether behavioral claims should be hedged. The module with the higher score — `LegalCautionModule` at 90 — takes precedence over `TechnicalDepthModule` at 70 in this conflict. The technical documentation patterns still contribute their non-conflicting directives — formal definitions, edge cases, invariants — but the legal caution directive prevails on the specific question of uncertainty language. The practitioner does not write conflict resolution code. They specify scores. The runtime resolves conflicts according to the score ordering. This is formal priority arbitration without custom arbitration logic, and it is one of the features that makes MODULE composition tractable at scale: a system with twenty modules does not require 190 pairwise conflict resolution rules — it requires twenty scores, and the runtime handles all pairwise conflicts by score comparison. The practitioner's responsibility is to assign scores that reflect the genuine priority ordering of the domains in their operational context.

The design of ACTIVATE_WHEN conditions requires care, because a poorly designed condition either activates a module too broadly — contributing irrelevant expertise to contexts where it adds noise — or too narrowly, failing to activate in cases where the module's expertise is genuinely needed. The canonical form of an ACTIVATE_WHEN condition is a logical expression over one or more fact store keys, using the standard comparison operators and logical connectives. Compound conditions with AND express contexts that require multiple simultaneous facts: a module that should activate only when a document is both high-priority and legal in domain should use `context.priority == 'high' AND context.domain == 'legal'`. Compound conditions with OR express contexts where any one of multiple conditions warrants activation: a module that should activate when either the domain is legal or the risk level is high should use `context.domain == 'legal' OR context.risk_level > 0.8`. Threshold conditions using `>`, `<`, `>=`, `<=` enable numeric activation criteria: `context.risk_level > 0.8` activates a risk-aware module only when the asserted risk level exceeds the specified threshold. The practitioner who designs ACTIVATE_WHEN conditions precisely is designing the expert system's activation policy from first principles — specifying exactly which operational contexts warrant each module's expertise.

Agicore's compile-time validation for MODULE declarations checks that ACTIVATE_WHEN conditions are syntactically valid logical expressions, that all fact store keys referenced in the condition are valid key identifiers, and that all score values are non-negative integers. The compiler does not validate the semantic appropriateness of the activation conditions — whether `context.domain == 'legal'` is the right condition for legal caution is a domain judgment that cannot be statically verified. But it catches the structural errors that would prevent the condition from being evaluated at runtime: malformed expressions, undefined operators, invalid key formats. This is the consistent principle of Agicore's compile-time validation: it enforces structural correctness without attempting to validate semantic appropriateness. The `.agi` file cannot contain syntactically invalid declarations; it can contain semantically inappropriate ones. The former class of errors is eliminated at compile time; the latter class is addressed through testing and operational review.

The synthesis of CHANNEL, TRIGGER, REASONER, SESSION, and MODULE produces a cognitive system that is qualitatively different from any system built from these components individually, and the nature of this difference deserves explicit statement. A CHANNEL alone is a typed message queue — useful for decoupling producers from consumers, but not capable of autonomous action. A TRIGGER alone is a reactive agent — capable of detecting conditions and firing actions, but only as capable as the actions it can fire. A REASONER alone is a structured inference specification — capable of producing auditable AI outputs, but only when explicitly invoked. A SESSION alone is a behavioral profile — a named configuration that must be activated by something. A MODULE alone is a knowledge bundle — expert expertise that must be integrated into inference context by something. When these five declaration types compose together through the channel contract and the reactive orchestration model, the emergent behavior is a **stateful, reactive, semantically typed, architecturally validated cognitive production system**: one that monitors its input channels continuously, responds to incoming work without polling, assembles structured inference contexts from the domain model, operates in formally specified behavioral modes, contributes specialized expertise automatically when conditions warrant, and produces a complete audit trail of every action. This is not an AI chatbot with extra steps. It is a cognitive production system in the engineering sense — one whose behavior is specified, validated, and improvable at the declaration level, not at the runtime level.

The gap between a raw API call and a system composed of these five declaration types is the gap that the ERP thesis describes. The cognitive ERP thesis — established in the early chapters of this book — holds that the REASONER, the SESSION, the MODULE, the TRIGGER, and the CHANNEL are to the cognitive production system what the general ledger, the materials requirement planning module, the workflow engine, the event processor, and the integration bus are to the enterprise resource planning system. SAP does not merely provide database tables. It provides a formal model of how an organization's resources flow, how processes are triggered, how approvals are governed, how exceptions are handled, and how the full operational history is recorded. The cognitive ERP provides the same governance framework for cognitive resources: how AI inference is structured, how operating modes are configured, how expert knowledge is activated, how reactive events are handled, and how the full inference history is recorded. An organization that manages its cognitive production with raw API calls is in the position of an organization that manages its supply chain with spreadsheets: it can get work done, but it cannot scale, cannot audit, cannot improve systematically, and cannot govern quality. The SESSION and MODULE declarations are two of the five governance mechanisms that distinguish the cognitive ERP from the spreadsheet.

The practitioner who builds with SESSION and MODULE is making a specific kind of commitment: a commitment to treating the system's behavioral configuration as an engineering artifact rather than a runtime variable. The implicit operating modes that practitioners carry in their heads — "I write more carefully when I'm doing legal work," "I need to be more specific on technical specs" — become SESSION and MODULE declarations: named, documented, validated, version-controlled, and improvable. The organizational expertise that accumulates in individual practitioners' judgments — "flag anything that sounds like liability," "always specify edge cases for API contracts" — becomes MODULE pattern lists: auditable knowledge that does not leave the organization when the practitioner does. This is the organizational learning dimension of the CSE practitioner's obligation to encode. The Jidoka loom stopped the thread break from propagating; the SESSION and MODULE stop the knowledge break from propagating — the loss of institutional expertise when expert practitioners are unavailable, reassigned, or departed. What is encoded in the `.agi` file persists. What exists only in the practitioner's head does not. The practitioner's highest obligation is to encode.

The testing implications of SESSION and MODULE declarations are more demanding than for CHANNEL, TRIGGER, or REASONER, and practitioners who reach for these declarations without a testing strategy will discover the hard way that behavioral configuration is difficult to validate in production. A CHANNEL can be tested by verifying that messages of the declared PACKET type can be published and retrieved. A TRIGGER can be tested by publishing a message and verifying that the designated FIRES action executes. A REASONER can be tested by running it against a known dataset and verifying that the output meets the quality criteria. But a SESSION's behavioral effect is inherently qualitative: does operating in `AnalyticalMode` actually produce more analytically rigorous outputs than operating with no session configuration? The answer requires a comparative evaluation — run the same REASONER with `AnalyticalMode` active and without it, evaluate both outputs against the analytical rigor criteria, and verify that the session-configured run scores materially higher. This is a behavioral test, not a structural test, and it requires human evaluation or a carefully designed evaluation REASONER. Similarly, a MODULE's activation condition can be tested structurally — does the module activate when `context.domain == 'legal'` is asserted? — but its substantive contribution requires behavioral evaluation: does the activated module's pattern list actually produce outputs that are more legally cautious, as measured by a QC REASONER designed to evaluate legal caution? The practitioner who treats SESSION and MODULE as configuration values that are set once and trusted indefinitely is accepting unvalidated behavioral risk. Both declaration types require periodic behavioral evaluation, preferably using a REASONER whose purpose is to evaluate the session or module's substantive effect.

The long-arc consequence of SESSION and MODULE design is the **behavioral specification corpus**: the growing collection of `.agi` declarations that formally encode an organization's cognitive production knowledge. Over time, as sessions are refined and modules are extended based on operational experience, the `.agi` file becomes a comprehensive specification of how the organization's cognitive production system thinks — what operating modes it can enter, what expert knowledge it can activate, under what conditions it activates each, and how it resolves conflicts among co-active expertise. This specification corpus is a form of institutional knowledge in the most precise sense: it is knowledge about how the organization processes information, made explicit, structured, and auditable. It can be reviewed by new practitioners to understand the system's operational intent. It can be compared across versions to understand how the system's behavior has evolved. It can be audited by compliance functions to verify that the system's behavioral boundaries are consistent with regulatory requirements. It can be tested against behavioral benchmarks to verify that the system's effectiveness has not degraded as new sessions and modules have been added. The `.agi` file is, in the end, the cognitive production system's specification document — not a diagram of its intended behavior but a complete, executable, validated description of its actual behavior. SESSION and MODULE are the declarations that make the system's behavioral sophistication legible in that document, converting what would otherwise be embedded knowledge in prompt strings and runtime variables into first-class architectural artifacts. This legibility is not merely convenient. It is the precondition for systematic improvement, and systematic improvement is the precondition for a cognitive production system that gets better rather than merely older.
