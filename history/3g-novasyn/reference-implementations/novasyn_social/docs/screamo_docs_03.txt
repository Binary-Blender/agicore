3.9 FEED SCANNING AND OPPORTUNITY COMMENTING

FR-9.1: The system shall retrieve the user’s LinkedIn feed via the LinkedIn API at defined intervals.
FR-9.2: The system shall analyze each post for topic alignment with the user’s known interests, expertise, and opinions.
FR-9.3: Posts that strongly align with user views shall trigger Agree-and-Amplify mode.
FR-9.4: Posts that conflict with user views but present an educational opportunity shall trigger Educate mode.
FR-9.5: Feed scanning subsystem shall generate a justification score for each proposed comment.
FR-9.6: The system shall provide draft comments for user approval before posting, unless SPC automation permits auto-posting for low-risk Agree-and-Amplify comments.
FR-9.7: The system shall respect platform rate limits and implement throttling when necessary.
FR-9.8: The system shall log all proposed opportunities, even those rejected by the user, for model refinement.

3.10 BATTLE MODE RESPONSE LOGIC

FR-10.1: Battle mode shall be triggered when a message is classified as hostile, confrontational, or directly challenging the user's statement.
FR-10.2: Battle mode shall generate a structured rebuttal including:

Identification of opposing claims

Logical deconstruction of each claim

Evidence-based counterpoints

Reinforcement of user stance

Invitation for civil discourse or closure statement

FR-10.3: Battle mode shall maintain strict professionalism and avoid personal attacks.
FR-10.4: Battle mode drafts shall require explicit human approval and shall never be auto-sent.
FR-10.5: Battle mode shall use a specialized AI model optimized for argument breakdown and clarity.
FR-10.6: The system shall store all battle mode drafts and user edits for analysis of argumentation patterns.

SYSTEM ARCHITECTURE

5.1 ARCHITECTURAL OVERVIEW
Screamo uses a service-oriented architecture composed of independent, loosely coupled microservices. Each subsystem runs as a containerized deployment. Communication occurs through internal REST APIs and asynchronous queues.

5.2 COMPONENT DESCRIPTIONS

Ingestion Service
Polls external APIs, normalizes incoming messages, stores them in the database.

Classification Service
Classifies messages using an LLM or fine-tuned model. Outputs classification metadata.

Prioritization Service
Computes and updates priority scores based on predefined algorithms and user feedback.

Draft Generation Service
Generates preliminary responses using a base LLM and applies style transformations.

Style Rewrite Engine
Uses RAG and embeddings to align tone with user’s writing.

Knowledge Base Service
Manages embeddings, style samples, redline topics, and opinion maps.

Feedback Processing Service
Analyzes user edits and updates preference models.

SPC Controller
Evaluates SPC metrics and determines allowable automation level.

Feed Scanner Service
Analyzes user’s LinkedIn feed and identifies comment opportunities.

Battle Mode Engine
Constructs structured argumentation responses.

UI Frontend
Handles user display and interaction.

Logging and Audit Service
Captures system events and stores them for compliance and review.

5.3 DATA FLOW DESCRIPTION

Flow sequence for inbound messages:

Platform → Ingestion Service

Normalization → Database

Classification Service → classification results stored

Prioritization Service → priority score stored

UI retrieves data → displays sorted inbox

User requests a draft → Draft Generation Service

User approves/edits → stored in feedback logs

Feedback Processing Service updates models

Flow sequence for feed scanning:

Feed Scanner fetches posts

Classification of posts

Opportunity detection

Draft Generation for comments

User approval

Posting via LinkedIn API

5.4 CONTROL FLOW DESCRIPTION

Automation levels are controlled by SPC Controller:
Manual → Assisted → Automatic (low-risk only)
Battle mode always remains Manual.
